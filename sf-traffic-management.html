Shared flow

sf-traffic-management

sf-traffic-management is the post-identity traffic gate. It runs two pure policies in a fixed order with no conditions: a per-client SpikeArrest (SA-PerClient, 10ps) that smooths instantaneous bursts, then a distributed, synchronous hourly Quota (Q-ByClient, 1000 per rolling hour) that enforces a hard consumption ceiling shared across every message processor in the region. Both are designed to key on the OAuth client_id from the validated token — which is why this flow must run after token validation — but as shipped both key on client.ip as a placeholder until that variable is reliably present. There is no JavaScript; the entire flow is two native policies wired by sharedflows/default.xml.

Where it sits

This is a quantitative gate, not a correctness gate: it answers "is this caller asking too fast or too often?", not "is this caller who they claim to be?". It deliberately runs late — after the access token has been validated — so that the rate-limit subject can graduate from a coarse network identity (client.ip) to a precise application identity (the token's client_id).

The bundle declares exactly two policies and a single default sub-flow:

NameTypePurpose
SA-PerClient SpikeArrest Smooth instantaneous bursts at 10ps per identifier. UseEffectiveCount makes the rate apply across the message-processor pool rather than per-MP. Keyed on client.ip today; intended to key on the token client_id in production.
Q-ByClient Quota (type="calendar") Enforce a hard ceiling of 1000 requests per 1 hour per identifier. Distributed + Synchronous keep one authoritative count consistent across all MPs in the region. Same identifier story as the spike arrest.
SpikeArrest is not a fractional-window quota

A SpikeArrest rate of 10ps does not mean "10 in any one-second window". Apigee converts the rate into a minimum inter-request interval — 10ps becomes one request roughly every 100 ms per identifier; a second request arriving 40 ms later is rejected even though the count for that second is only two. Treat it as a smoother, and use the Quota for volume.

Step order and conditions

The default sub-flow runs both steps top to bottom and neither step carries a Condition — every request that reaches this flow is metered, first for burstiness, then for cumulative volume. SpikeArrest comes first on purpose: it is the cheapest possible rejection, so a flood is shed before it can churn the distributed quota counter.

Smooth the burst — SA-PerClient (unconditional)

The SpikeArrest evaluates first. If requests for a given identifier arrive faster than the 10ps-derived interval, the offending request is rejected immediately with a SpikeArrestViolation fault. Because continueOnError="false", that rejection short-circuits the flow — Q-ByClient never runs for a throttled request, so a spike cannot inflate the hourly count.

Enforce the volume ceiling — Q-ByClient (unconditional)

Requests that pass the smoother hit the calendar Quota. The distributed counter is incremented synchronously; when an identifier exceeds 1000 within the current hourly bucket the policy raises QuotaViolation. Both faults are caught and re-shaped downstream by sf-error-handling into the OBIE envelope with the right 429 status.

sharedflows/default.xmlXML
<SharedFlow name="default">
  <!-- 1. Cheap burst smoother runs first; unconditional. -->
  <Step>
    <Name>SA-PerClient</Name>
  </Step>
  <!-- 2. Hard hourly ceiling; only reached if the smoother let the request through. -->
  <Step>
    <Name>Q-ByClient</Name>
  </Step>
</SharedFlow>

The two policies in full

Both policies are short and native. The comment blocks in the source are load-bearing documentation of the intended production identifier, so they are reproduced here verbatim.

policies/SA-PerClient.xmlXML
<SpikeArrest async="false" continueOnError="false" enabled="true" name="SA-PerClient">
  <DisplayName>SA-PerClient</DisplayName>
  <Properties/>
  <Rate>10ps</Rate>
  <!-- production should key on the token client_id once available -->
  <Identifier ref="client.ip"/>
  <UseEffectiveCount>true</UseEffectiveCount>
</SpikeArrest>
policies/Q-ByClient.xmlXML
<Quota name="Q-ByClient" type="calendar">
  <DisplayName>Q-ByClient</DisplayName>
  <Allow count="1000"/>
  <Interval>1</Interval>
  <TimeUnit>hour</TimeUnit>
  <!-- production keys on client_id -->
  <Identifier ref="client.ip"/>
  <Distributed>true</Distributed>
  <Synchronous>true</Synchronous>
  <StartTime>2026-01-01 00:00:00</StartTime>
</Quota>

Why these specific settings

UseEffectiveCount = true

Apigee distributes a SpikeArrest budget across the message-processor pool. With UseEffectiveCount on, 10ps is the rate the client experiences regardless of how many MPs are serving traffic; without it, each MP would enforce 10ps independently and a caller hitting N MPs could burst at up to N × 10ps.

Distributed + Synchronous

A FAPI quota must be a real ceiling, not a per-MP approximation. Distributed="true" keeps a single shared counter; Synchronous="true" reads and writes it on every request so the limit is exact. The trade is a small per-request latency cost against the counter store — acceptable for the security posture an ASPSP needs.

type="calendar" + StartTime

A calendar quota needs an explicit StartTime anchor; here 2026-01-01 00:00:00. From that anchor Apigee derives fixed, repeating hourly buckets, so every client's window resets on the same clock boundary rather than drifting per first-call time.

10ps vs 1000/hour

The two limits are intentionally non-overlapping concerns. 10ps caps shape (no hammering); 1000/hour caps volume (fair share over time). A well-behaved client pacing under 10ps can still be stopped by the hourly ceiling, and vice-versa.

FAPI relevance

FAPI 1.0 Advanced raises the bar on availability and abuse-resistance for high-value financial APIs, and the UK Open Banking operational guidelines expect ASPSPs to rate-limit per registered TPP. This flow is where that expectation is enforced:

  • Per-client fairness, not per-IP. The whole reason the flow runs post-identity is so that the meter subject is the registered client application (client_id), giving each TPP its own independent budget. A shared NAT or a TPP fronted by many IPs should still see one consistent limit — which is exactly what client.ip keying cannot deliver, hence the production swap below.
  • Defence in depth. Throttling sits after authentication so an attacker cannot exhaust a victim's quota with unauthenticated traffic; spend is only ever attributed to a validated identity.
  • Standards-shaped rejections. A throttled request must come back as 429 Too Many Requests in the OBIE error envelope. This flow only raises the native SpikeArrestViolation / QuotaViolation faults; sf-error-handling maps them to the correct status and body, and sf-logging records the breach with the interaction id minted by sf-fapi-headers.
As-shipped, both policies key on client.ip — fix before production

The single most important change in this flow is the <Identifier>. Keying on client.ip means a TPP behind a single egress IP can be throttled by unrelated traffic from that IP, and a TPP spread across many IPs gets many budgets — both violate per-client fairness. The source comments are explicit: switch both identifiers to the validated token's client id (for example <Identifier ref="apigee.client_id"/>, or whatever variable your token validation flow publishes) once that value is reliably present.

Configuration knobs for production

This flow is almost entirely defaults and one critical placeholder. The table below is the checklist to confirm before promoting beyond the eval environment.

KnobWhereDefault & production guidance
Identifier (both) SA-PerClient & Q-ByClient <Identifier ref> Placeholder. Ships as client.ip. Change both to the OAuth client_id from the validated token (e.g. apigee.client_id or your token flow's variable) so budgets are per registered TPP.
Spike rate SA-PerClient <Rate> Default 10ps. Set to your per-TPP burst tolerance; remember it is an inter-request interval, not a per-second bucket. pm / ps units are supported.
Quota allowance Q-ByClient <Allow count> Default 1000. Set per TPP tier. For per-product or per-app variation, drive this from a variable (<Allow count="1000" countRef="verifyapikey.quota.count"/>) sourced from an API product or KVM.
Quota window Q-ByClient <Interval> / <TimeUnit> Default 1 × hour. Adjust to match your published rate-limit policy; type="calendar" keeps windows aligned to StartTime.
Calendar anchor Q-ByClient <StartTime> 2026-01-01 00:00:00. Any past timestamp works as the bucket origin; keep it stable, since changing it re-phases every client's window.
Effective-count scope SA-PerClient <UseEffectiveCount> Default true (pool-wide rate). Leave on for accurate global throttling; only disable if you deliberately want per-MP enforcement.

Contract: what callers and downstream get

Always true after this flow

Every request that proceeds past this flow has been admitted by both the burst smoother and the hourly ceiling for its identifier. Downstream policies and the backend can assume the request is within rate.

What it never does

It performs no authentication, sets no headers, runs no script, and adds no conditions — it cannot let a request "skip" metering. A throttled request stops here with a native violation fault and is shaped to 429 by sf-error-handling.
