sf-preflight

sf-preflight is the outermost guard in the request path: it spends the cheapest possible work to reject abusive or malformed traffic before any header parsing, token validation, or crypto runs. One SpikeArrest caps per-client rate, then three fail-fast RaiseFault guards short-circuit unsupported HTTP methods (405), unsupported media types (415), and oversized payloads (413). Every rejection is emitted as an OBIE Read/Write JSON error carrying the caller's x-fapi-interaction-id.

In a FAPI 1.0 Advanced chassis the expensive layers — mTLS binding, JWS/JWE, JWT validation — must never be the first thing an attacker can force you to execute. sf-preflight exists so that a flood of garbage requests is shed at the door, deterministically and without allocating a parser or a key.

Policies

Four policies, declared in sf-preflight.xml and sequenced by sharedflows/default.xml. The bundle ships at ConfigurationVersion 4.0 and carries no resources.

NameTypePurpose
SA-SpikeArrest SpikeArrest Smooth per-client rate cap of 30ps, keyed on client.ip, counted cluster-wide. Always runs first; protects every downstream policy and the backend.
RF-MethodNotAllowed RaiseFault Returns 405 when request.verb is outside the allowed set (GET, POST, PUT, DELETE, PATCH).
RF-UnsupportedMediaType RaiseFault Returns 415 when a body-bearing verb (POST/PUT/PATCH) arrives without an application/json or application/jose content type.
RF-PayloadTooLarge RaiseFault Returns 413 when Content-Length exceeds 1 MiB (1 048 576 bytes).

Step order and conditions

Steps are ordered cheapest-first and most-protective-first. The rate limiter is unconditional; the three guards are guarded by Apigee condition expressions so they fire only on the request shapes they own.

SA-SpikeArrest — unconditional

No <Condition>. Runs on every request as the first step. If the per-IP rate is exceeded the policy itself raises SpikeArrestViolation and the request is rejected before any guard is evaluated.

RF-MethodNotAllowed — when verb not in allow-list

Condition: !(request.verb = "GET" or … or request.verb = "PATCH"). Any other verb (HEAD, OPTIONS, TRACE, CONNECT, or junk) is rejected with 405.

RF-UnsupportedMediaType — when body verb lacks JSON/JOSE

Condition fires only for POST/PUT/PATCH and only when content-type is neither exactly application/json, nor matches application/json;* (charset params), nor matches application/jose* (covers application/jose+json). Otherwise 415.

RF-PayloadTooLarge — when Content-Length > 1 MiB

Condition: content-length != null and content-length GreaterThan 1048576. Rejects with 413 before the body is buffered for any downstream parsing or signature check.

The size guard has a deliberate gap

The 413 condition requires content-length != null. A chunked or streamed request that omits Content-Length will not be caught here — enforce a hard ceiling at the load balancer / message-processor I/O limits as defence in depth. sf-preflight rejects the cheap, declared-oversize case; it is not the only line.

The flow definition

This is the complete sequencing in sharedflows/default.xml — the canonical source of step order and conditions.

sharedflows/default.xmlXML
<SharedFlow name="default">
  <!-- 1. Spike arrest is the outermost guard and always runs, keyed by client IP. -->
  <Step><Name>SA-SpikeArrest</Name></Step>

  <!-- 2. Reject any HTTP verb outside the supported set with 405. -->
  <Step>
    <Name>RF-MethodNotAllowed</Name>
    <Condition>!(request.verb = "GET" or request.verb = "POST" or request.verb = "PUT" or request.verb = "DELETE" or request.verb = "PATCH")</Condition>
  </Step>

  <!-- 3. For body-bearing verbs, require a JSON or JOSE content type; otherwise 415. -->
  <Step>
    <Name>RF-UnsupportedMediaType</Name>
    <Condition>(request.verb = "POST" or request.verb = "PUT" or request.verb = "PATCH") and (request.header.content-type != "application/json") and (request.header.content-type !~ "application/json;*") and (request.header.content-type !~ "application/jose*")</Condition>
  </Step>

  <!-- 4. Reject oversized payloads (> 1 MiB) before buffering downstream work. -->
  <Step>
    <Name>RF-PayloadTooLarge</Name>
    <Condition>(request.header.content-length != null) and (request.header.content-length GreaterThan 1048576)</Condition>
  </Step>
</SharedFlow>

The rate limiter

The only stateful policy in the bundle. 30ps is enforced as smoothing — roughly one request every 33 ms per identifier — rather than a 30-in-a-burst window, so a client cannot spend its whole budget in the first millisecond of a second.

policies/SA-SpikeArrest.xmlXML
<SpikeArrest name="SA-SpikeArrest">
  <DisplayName>SA-SpikeArrest</DisplayName>
  <!-- Smooth per-client rate cap. 30ps is enforced as ~1 request / 33ms per IP. -->
  <Rate>30ps</Rate>
  <Identifier ref="client.ip"/>
  <!-- Count across all message processors so the cap holds cluster-wide. -->
  <UseEffectiveCount>true</UseEffectiveCount>
</SpikeArrest>
client.ip is only as true as your edge

UseEffectiveCount=true makes the cap hold across all message processors instead of per-MP. But behind a load balancer, client.ip can resolve to the LB egress rather than the real caller — collapsing every client into one bucket. In production, terminate the proxy chain correctly and key the identifier on the trusted forwarded client address (e.g. X-Forwarded-For) where applicable.

Error responses

All three guards are RaiseFault policies with IgnoreUnresolvedVariables=true, so a missing x-fapi-interaction-id never breaks fault assembly — the Id field is simply left empty. Each fault sets a JSON body in the OBIE Read/Write API error shape: Code, Id, Message, and an Errors[] array with an OBIE ErrorCode.

StatusOBIE ErrorCodeTrigger
405UK.OBIE.UnsupportedHttpMethodVerb outside GET/POST/PUT/DELETE/PATCH
415UK.OBIE.UnsupportedMediaTypeBody verb without application/json or application/jose
413UK.OBIE.Field.InvalidContent-Length over 1 MiB
policies/RF-MethodNotAllowed.xmlXML
<RaiseFault name="RF-MethodNotAllowed">
  <DisplayName>RF-MethodNotAllowed</DisplayName>
  <IgnoreUnresolvedVariables>true</IgnoreUnresolvedVariables>
  <FaultResponse>
    <Set>
      <StatusCode>405</StatusCode>
      <ReasonPhrase>Method Not Allowed</ReasonPhrase>
      <Payload contentType="application/json">{"Code":"405",
        "Id":"{request.header.x-fapi-interaction-id}",
        "Message":"Method Not Allowed",
        "Errors":[{"ErrorCode":"UK.OBIE.UnsupportedHttpMethod",
        "Message":"The HTTP method is not supported for this resource."}]}</Payload>
    </Set>
  </FaultResponse>
</RaiseFault>

FAPI relevance

This flow is not a FAPI protocol control in itself — it does no token or signature work. Its job is to make the FAPI layers that follow safe, cheap, and predictable.

Defence in depth Attack-surface reduction Availability JOSE-aware

Fail fast, before crypto

A request that will be rejected anyway is rejected here — before sf-mtls-binding, sf-jws-validation, or sf-jwe spend CPU on certificates and keys. That keeps the expensive cryptographic path off the abuse hot-loop.

JOSE content types are first-class

FAPI request signing and encryption travel as application/jose / application/jose+json. The 415 guard explicitly allows them so signed and encrypted request objects pass preflight while plain text or form bodies are turned away.

Bounded resource use

The spike cap and the 1 MiB payload ceiling bound how much an unauthenticated caller can cost you — protecting availability for the ASPSP estate, an explicit FAPI operational concern.

Consistent OBIE errors from byte zero

Even the earliest rejections speak the OBIE error contract and echo x-fapi-interaction-id, so TPP-side correlation and error handling work identically across every rejection point.

Configuration knobs

The values below are the placeholders to review per environment before production. None require code changes beyond the bundle XML.

KnobFile · elementShipped value & production note
Rate cap SA-SpikeArrest.xml · <Rate> 30ps — size to your per-client SLA. Use ps for sub-second smoothing; pm allows longer bursts.
Rate identifier SA-SpikeArrest.xml · <Identifier ref> client.ip — switch to a trusted forwarded address behind an LB, or to an app/key identifier for per-TPP fairness.
Cluster counting SA-SpikeArrest.xml · <UseEffectiveCount> true — keep on for a global cap; set false only if you intend a per-MP limit.
Allowed verbs default.xml · RF-MethodNotAllowed <Condition> GET/POST/PUT/DELETE/PATCH — trim to only the verbs a given proxy actually serves.
Allowed media types default.xml · RF-UnsupportedMediaType <Condition> application/json + application/jose* — extend only if a downstream genuinely needs another type.
Max payload default.xml · RF-PayloadTooLarge <Condition> 1048576 (1 MiB) — lower for read-heavy APIs; pair with LB-level limits for chunked requests.
Place this flow first, every time

sf-preflight only delivers its guarantees if it is the first shared flow in the proxy PreFlow, ahead of header, mTLS, and JWS/JWE flows. Mount it later and you have already paid for the work it exists to avoid. See the reference proxy for the canonical PreFlow ordering.
