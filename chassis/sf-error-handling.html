sf-error-handling

sf-error-handling is the proxy's single fault exit. It is invoked from the DefaultFaultRule (via the FC-ErrorHandling FlowCallout) whenever any policy raises a fault, and it turns whatever broke internally into one standardized UK Open Banking Read/Write error body. Two policies do the work: JS-MapError runs errorMap.js to classify the fault — first by Apigee fault.name, then by originating HTTP status — into a UK.OBIE.* code, an HTTP status, and a non-leaking message; AM-ErrorResponse serialises those into the OBIE JSON shape { Code, Id, Message, Errors[] }. A hard rule guarantees that for any status ≥ 500 the caller only ever sees UK.OBIE.UnexpectedError and a fixed generic message — internal detail never leaks.

In a FAPI 1.0 Advanced chassis the fault path is part of the security surface. A stack trace, a backend hostname, or a policy name echoed in a 500 body is an information-disclosure finding. This flow exists so that every failure — from every layer — collapses into one predictable, machine-parseable contract that a TPP can handle, while server-side detail stays server-side.

How it is invoked

The bundle is not mounted in a PreFlow. It is wired into the proxy's DefaultFaultRule, so it runs only when a fault is raised anywhere in the request, response, or target flow. The proxy reaches it through a one-line FlowCallout.

apiproxy/proxies/default.xmlXML
<DefaultFaultRule name="default-fault">
  <Step>
    <Name>FC-ErrorHandling</Name>
  </Step>
</DefaultFaultRule>
apiproxy/policies/FC-ErrorHandling.xmlXML
<FlowCallout name="FC-ErrorHandling">
  <DisplayName>FC-ErrorHandling</DisplayName>
  <SharedFlowBundle>sf-error-handling</SharedFlowBundle>
</FlowCallout>
This is the catch-all, not the only error path

The preflight guards (sf-preflight) build their own OBIE bodies inline with RaiseFault and short-circuit before this flow. sf-error-handling is the safety net for everything else — token, signature, schema, quota, and target faults — so that any fault that bubbles up to the DefaultFaultRule still exits in the OBIE shape.

Policies

Two policies, declared in sf-error-handling.xml and sequenced by sharedflows/default.xml. The bundle ships at ConfigurationVersion 4.0 and carries one resource, jsc://errorMap.js.

NameTypePurpose
JS-MapError Javascript Runs errorMap.js. Reads fault.name, error.status.code, and message.status.code; classifies into an HTTP status, a UK.OBIE.* code, and a safe message; publishes ob.http.status, ob.error.code, ob.error.message. timeLimit=5000, continueOnError=true.
AM-ErrorResponse AssignMessage Overwrites the response (AssignTo type="response", createNew=false): sets the HTTP status from ob.http.status and writes the OBIE JSON body { Code, Id, Message, Errors[] }. IgnoreUnresolvedVariables=true.

Step order and conditions

Two steps, no conditions — both always run, in order. Classification must precede serialisation because AM-ErrorResponse reads the three ob.* variables that JS-MapError sets.

JS-MapError — unconditional

Executes errorMap.js. Gathers the fault signals, classifies them, and writes ob.http.status, ob.error.code, and ob.error.message. continueOnError=true means that even a script error does not derail the fault rule — the policy's own try/catch still publishes a safe 500 first.

AM-ErrorResponse — unconditional

Reads the three ob.* variables plus fapi.interaction.id and assembles the OBIE error response, replacing whatever partial response the failing flow had produced. With IgnoreUnresolvedVariables=true, a missing interaction id leaves Id empty rather than breaking assembly.

sharedflows/default.xmlXML
<SharedFlow name="default">
  <Step><Name>JS-MapError</Name></Step>
  <Step><Name>AM-ErrorResponse</Name></Step>
</SharedFlow>

Classification: errorMap.js

The script reads three signals and resolves them to a triple. It starts from a safe default and refines in a fixed precedence: fault name first, originating status code second.

in: fault.name in: error.status.code in: message.status.code out: ob.http.status out: ob.error.code out: ob.error.message

The originating status is firstNonNull(error.status.code, message.status.code) — error.status.code wins, message.status.code is the fallback, and it may be null. The starting default is 500 / UK.OBIE.UnexpectedError / "An unexpected error occurred".

Pass 1 — by fault name

The fault name is lower-cased and matched by substring, so a single check covers many policy-specific fault names. The branches are evaluated top-down and the first match wins — order matters: the broad invalid 400-branch sits last, so invalidaccesstoken and invalidapikey are caught by the 401-branch above it.

insufficient, forbidden, scope (not quota/spike) 403 UK.OBIE.Resource.Forbidden
schemavalidation, extractvariable, invalidvariable, jsonpath, jsonthreatprotection, xmlthreatprotection, badrequest, invalid 400 UK.OBIE.Field.Invalid

The 429 / 403 branch is nested

The throttling and forbidden cases share one outer branch matched by insufficient | forbidden | scope | quotaviolation | spikearrest. Inside it, a second test splits quotaviolation / spikearrest off to 429 UK.OBIE.Rate.Limit; everything else in the branch becomes 403 UK.OBIE.Resource.Forbidden.

Pass 2 — by status code

This pass runs only when pass 1 left the result at the unexpected default and a status code is present (errorCode === 'UK.OBIE.UnexpectedError' && rawStatus !== null). A fault name that already classified is never overridden by the status.

The 5xx leak-masking

Suppression of internal detail on server errors is defended in three independent layers — any one of them alone is sufficient.

Layer 1 — generic by construction

The unexpected default is the 5xx case: 500 / UK.OBIE.UnexpectedError / "An unexpected error occurred". An unclassifiable fault never falls through to anything more specific.

Layer 2 — hard safety net

After both classification passes, an unconditional final guard re-asserts the generic code and message for any resolved status >= 500, regardless of how it was reached. A 5xx can never carry a specific code or message.

Layer 3 — catch-all fallback

The entire body is wrapped in try/catch. If the script itself throws, the catch publishes a safe 500 / UK.OBIE.UnexpectedError / generic message, so the fault rule still emits a well-formed OBIE error.

resources/jsc/errorMap.jsJavaScript
// --- Hard safety net: never leak internals on 5xx --------------------
if (httpStatus >= 500) {
  errorCode = 'UK.OBIE.UnexpectedError';
  errorMessage = GENERIC_5XX_MESSAGE;
}

context.setVariable('ob.http.status', httpStatus);
context.setVariable('ob.error.code', errorCode);
context.setVariable('ob.error.message', errorMessage);

} catch (e) {
  // Absolute fallback: safe, generic, well-formed.
  context.setVariable('ob.http.status', 500);
  context.setVariable('ob.error.code', 'UK.OBIE.UnexpectedError');
  context.setVariable('ob.error.message', GENERIC_5XX_MESSAGE);
}
Never widen the 5xx message

The generic message and the hard safety net are the whole point of this flow. Do not be tempted to surface error.message, fault.cause, or a target body on a 5xx for "easier debugging" — that is an information-disclosure regression. Carry the detail in sf-logging instead, correlated by the same interaction id.

The OB error JSON shape

AM-ErrorResponse sets the HTTP status from ob.http.status and writes the UK Open Banking Read/Write error body. The interaction id surfaces as Id; the message and OBIE code come from the script.

policies/AM-ErrorResponse.xmlXML
<AssignMessage name="AM-ErrorResponse" continueOnError="false" enabled="true">
  <IgnoreUnresolvedVariables>true</IgnoreUnresolvedVariables>
  <AssignTo createNew="false" type="response"/>
  <Set>
    <StatusCode>{ob.http.status}</StatusCode>
    <Payload contentType="application/json">{
  "Code": "{ob.http.status}",
  "Id": "{fapi.interaction.id}",
  "Message": "{ob.error.message}",
  "Errors": [
    {
      "ErrorCode": "{ob.error.code}",
      "Message": "{ob.error.message}"
    }
  ]
}</Payload>
    <Headers>
      <Header name="Content-Type">application/json</Header>
    </Set>
  </Set>
</AssignMessage>
FieldSourceMeaning
Code{ob.http.status}Top-level code. In this chassis it carries the HTTP status as a string (e.g. "401") rather than an OBIE category label.
Id{fapi.interaction.id}The x-fapi-interaction-id for this request, set upstream by sf-fapi-headers. Lets a TPP correlate the error with its request and with server logs.
Message{ob.error.message}Human-readable, non-leaking message. Generic for 5xx; specific (but still safe) for handled 4xx.
Errors[].ErrorCode{ob.error.code}The UK.OBIE.* machine code a TPP branches on — e.g. UK.OBIE.Unauthorized, UK.OBIE.Field.Invalid.
Errors[].Message{ob.error.message}Per-error message; mirrors the top-level Message in this implementation.
"Code" carries the HTTP status, not an OBIE category

The strict OBIE Read/Write schema expects the top-level Code to be a high-level category string. Here it is populated with {ob.http.status} — the numeric HTTP status as text. The machine-actionable UK.OBIE.* value lives in Errors[].ErrorCode. If you are certifying against a strict OBIE validator, change the Code binding in AM-ErrorResponse to emit a category code (or a fixed "OBIE" banner) rather than the status.

FAPI relevance

This flow does no token or crypto work, but it is a FAPI control: the spec's threat model treats verbose errors as an attack aid, and consistent, correlatable errors as an operational requirement for the ASPSP estate.

Information-disclosure control OBIE error contract Interaction-id correlation Fail-closed

No internal leakage on 5xx

Three independent layers force every server error to a fixed generic message and UK.OBIE.UnexpectedError. Backend hostnames, stack traces, and policy names never reach the TPP.

One contract for every fault

Token, signature, schema, quota, and target faults all exit in the same { Code, Id, Message, Errors[] } shape, so a TPP writes one error handler that works at every rejection point.

Correlatable by design

Id echoes x-fapi-interaction-id, tying the client-visible error to the server-side log record for the same request — debuggability without disclosure.

Fail-closed, always well-formed

continueOnError, the script try/catch, and IgnoreUnresolvedVariables mean the flow always emits a valid OBIE error — even when the classifier or a context variable misbehaves.

The placeholders to review per environment before production. All live in the bundle; none require touching the proxy.
