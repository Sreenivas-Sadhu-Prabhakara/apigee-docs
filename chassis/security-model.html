FAPI Security Model

Apigee Docs implements FAPI 1.0 Advanced for UK Open Banking as five composable, independently testable controls — one Apigee shared flow each. Together they cover the four FAPI guarantees (caller authentication, message integrity, message confidentiality, and token-to-sender binding) plus the mandated traceability headers:

  • Caller & authorization — sf-jwt-validation verifies the OAuth2 access-token JWT (RS256/PS256) against a rotating JWKS and enforces the accounts scope.
  • Request integrity / non-repudiation — sf-jws-validation verifies a detached JWS (PS256, b64=false per RFC 7797) over the body, with the OB critical headers.
  • Payload confidentiality — sf-jwe decrypts inbound (and can encrypt outbound) JWE using RSA-OAEP-256 + A256GCM.
  • Sender-constrained tokens — sf-mtls-binding matches the token's cnf.x5t#S256 against the SHA-256 thumbprint of the presented client mTLS certificate (RFC 8705).
  • Traceability — sf-fapi-headers validates and normalises the x-fapi-* headers.

Each layer fails closed and renders a UK OBIE error envelope. Every cryptographic anchor (JWKS URIs, issuer, audience, KVM key names) is a placeholder meant to be repointed per environment.

This page is the cross-cutting security narrative: what each control proves, where it is enforced, the actual algorithm and step order in code, and the threat it removes. The per-flow pages carry the full policy walkthroughs.

Defence in depth, layer by layer

The controls are ordered so each one builds on the variables the previous one published. sf-fapi-headers establishes the interaction id used in every fault; sf-jwt-validation verifies the token and exposes its claims (including cnf); sf-mtls-binding then consumes jwt.VJWT-AccessToken.claim.cnf, so it must run after JWT validation.

Traceability — sf-fapi-headers

Validate x-fapi-interaction-id (UUID) and x-fapi-customer-ip-address (IPv4/IPv6); echo or mint the interaction id and publish it on fapi.interaction.id for every downstream fault.

Authentication & authorization — sf-jwt-validation

Extract the bearer token, fault 401 if absent, verify signature/issuer/audience/expiry via JWKS, fault 403 without the accounts scope. Publishes all token claims.

Sender constraint — sf-mtls-binding

Recompute base64url(SHA-256(DER)) of the client certificate and compare to the token's cnf.x5t#S256. Fail closed; 401 on mismatch.

Request integrity — sf-jws-validation

Verify the detached JWS in x-jws-signature over request.content (b64=false), enforcing the OB critical headers.

Payload confidentiality — sf-jwe

If the request arrives as application/jose/application/jwe, decrypt it (RSA-OAEP-256 + A256GCM) before the target sees it; the matching encrypt policy is available for responses.

1 · OAuth2 JWT access tokens

Enforced in sf-jwt-validation

Proves the caller holds a valid, unexpired access token issued by our authorization server, for this API, carrying the right scope — without a network round-trip, because the token is a signed JWT verified against the rotating JWKS.

NameTypePurpose
EV-BearerTokenExtractVariablesPull the token after Bearer in the Authorization header into oauth.access.token.
RF-MissingTokenRaiseFault401 with WWW-Authenticate: invalid_token when no token is presented.
VJWT-AccessTokenVerifyJWTVerify signature (RS256/PS256), issuer, audience and expiry against the JWKS.
RF-InsufficientScopeRaiseFault403 with WWW-Authenticate: insufficient_scope when the accounts scope is absent.

Step order & conditions

The four steps run unconditionally except for the two faults, which are gated on flow variables:

  • EV-BearerToken — always; IgnoreUnresolvedVariables=true so an absent header does not fault here.
  • RF-MissingToken — when (oauth.access.token = null) or (oauth.access.token = "").
  • VJWT-AccessToken — always (only reached with a token present).
  • RF-InsufficientScope — when NOT(jwt.VJWT-AccessToken.claim.scope ~ "*accounts*").
policies/VJWT-AccessToken.xmlXML
<VerifyJWT name="VJWT-AccessToken">
  <Algorithm>RS256</Algorithm>          <!-- PS256 is the FAPI-preferred variant -->
  <Source>oauth.access.token</Source>
  <PublicKey>
    <JWKS uri="https://auth.example.com/.well-known/jwks.json"/>
  </PublicKey>
  <Issuer>https://auth.example.com</Issuer>
  <Audience>accounts-api</Audience>
</VerifyJWT>
FAPI relevance & threat

Signature verification defeats forged or tampered tokens. The Issuer check rejects tokens minted by a foreign AS; the Audience check stops a valid token for another API being replayed here; the scope gate enforces least privilege. Using a JWKS URI (not an inline key) lets the AS rotate signing keys without redeploying the proxy.

Configuration knobs

  • <Algorithm> — ships as RS256; switch to PS256 (RSASSA-PSS) to enforce the FAPI-preferred algorithm.
  • <JWKS uri>, <Issuer>, <Audience> — repoint at your real authorization server per environment.
  • The scope match string "*accounts*" — tighten to the exact OB scope your resource requires.

2 · Detached JWS request signatures

Enforced in sf-jws-validation

Proves the request body was authored by the TPP and reached us byte-for-byte intact — the basis for non-repudiation of writes and payments. The signature is detached (RFC 7797, b64=false) so the body is signed in place rather than re-encoded.

NameTypePurpose
VJWS-RequestVerifyJWSVerify the detached JWS in x-jws-signature over request.content using PS256 and the signing JWKS, enforcing OB critical headers.

Step order & conditions

A single step, gated on the signature header being present:

  • VJWS-Request — when (request.header.x-jws-signature != null). This lets the demo GET (no body, no signature) pass; any signature that is supplied is fully verified.
policies/VJWS-Request.xmlXML
<VerifyJWS name="VJWS-Request">
  <Algorithm>PS256</Algorithm>
  <Source>request.header.x-jws-signature</Source>
  <DetachedContent>request.content</DetachedContent>
  <PublicKey>
    <JWKS uri="https://auth.example.com/.well-known/signing-jwks.json"/>
  </PublicKey>
  <CriticalHeaders>b64,http://openbanking.org.uk/iat,http://openbanking.org.uk/iss,http://openbanking.org.uk/tan</CriticalHeaders>
</VerifyJWS>

The four critical (crit) headers are exactly those the OB Read/Write spec mandates:

b64 http://openbanking.org.uk/iat http://openbanking.org.uk/iss http://openbanking.org.uk/tan

b64=false selects detached, unencoded-payload signing; iat is the signing timestamp, iss the signer, and tan the trust anchor (the directory that vouches for the signing key). A verifier that does not understand a listed crit header must reject the JWS.

FAPI relevance & threat

The detached signature defeats request tampering by anything between the TPP and the resource (including a compromised TLS-terminating hop) and provides non-repudiation: the TPP cannot later deny a signed payment instruction. Pinning PS256 and the crit set blocks algorithm-downgrade and crit-stripping attacks.

Production hardening

On write/payment proxies, enforce presence: drop the x-jws-signature != null condition (or invert it) so a missing signature is rejected, not skipped. Point <JWKS uri> at the TPP/OB signing key set, sourced from a KVM per environment.

3 · JWE payload encryption

Enforced in sf-jwe

Adds application-layer confidentiality on top of TLS, so sensitive payloads stay encrypted past the TLS terminator — through load balancers, proxies, and logs — until the bank's own policy decrypts them. Uses RSA-OAEP-256 key wrapping with A256GCM authenticated content encryption.

NameTypePurpose
JS-DecryptJWEJavaScriptDecrypt the inbound request payload (jwe.mode=decrypt) back into request.content.
JS-EncryptJWEJavaScriptEncrypt an outbound response (jwe.mode=encrypt); intentionally un-stepped — attach on a proxy's response flow when needed.

Both policies share one resource, jsc://jwe.js, which uses the jsrsasign library (KJUR.jose.JWE). Direction is selected by the policy <Property name="jwe.mode">.

Step order & conditions

  • JS-DecryptJWE — when (request.header.content-type ~ "application/jose*") or (request.header.content-type ~ "application/jwe*").
  • JS-EncryptJWE — not stepped in default.xml; documented for response-side use.
resources/jsc/jwe.js — decrypt pathJavaScript
// Restrict accepted algorithms to the FAPI profile to avoid alg-downgrade.
var result = KJUR.jose.JWE.decrypt(payload, prvKey, {
  alg: ["RSA-OAEP-256"],
  enc: ["A256GCM"]
});
context.setVariable("request.content", result);

Implementation details that matter for security:

  • Algorithm allow-list. Both alg and enc are pinned, so a token offering a weaker algorithm is refused rather than honoured.
  • Shape check. isCompactJWE() requires exactly five base64url dot-segments before any crypto is attempted; non-JWE bodies pass through untouched.
  • Keys from KVM. The recipient private key is read from private.jwe.key (and the client public key from public.jwe.key) — loaded by a preceding KeyValueMapOperations policy from an encrypted KVM. The in-file REPLACE_WITH_KVM_VALUE_* strings are deliberate, loud placeholders.
  • Fail soft, never throw. Any error is recorded in jwe.error and the message left intact, so the error-handling flow can render an OB envelope rather than leaking a stack trace.
FAPI relevance & threat

JWE removes the assumption that "TLS is enough". It defends against payload disclosure at every point after TLS termination — mirrored traffic, debug logs, a breached intermediary. A256GCM is authenticated, so tampering with ciphertext fails the GCM tag; the pinned alg/enc list blocks downgrade.

Configuration knobs

  • private.jwe.key / public.jwe.key — populate from an encrypted KVM before this policy runs; never commit real keys.
  • jwe.kid — optional key id stamped into the JOSE header on encrypt so the recipient selects the right key.
  • timeLimit="5000" — JavaScript execution budget; tune for your payload sizes.

4 · Certificate-bound (sender-constrained) tokens

Enforced in sf-mtls-binding

Turns a bearer token into a holder-of-key token: it is only usable by the client that owns the mTLS certificate it was issued against. Implements RFC 8705 by recomputing the certificate thumbprint and comparing it to the token's cnf.x5t#S256 confirmation claim.

NameTypePurpose
JS-VerifyCertBindingJavaScriptCompute base64url(SHA-256(DER)) of the client cert and compare to the token's cnf.x5t#S256; publish cert.binding.match.
RF-CertBindingMismatchRaiseFault401 OB envelope when the thumbprints do not match.

Step order & conditions

  • JS-VerifyCertBinding — when (jwt.VJWT-AccessToken.claim.cnf != null) or (request.header.x-client-cert != null). This requires sf-jwt-validation to have run first to expose the cnf claim.
  • RF-CertBindingMismatch — when (cert.binding.match = false).
resources/jsc/certBinding.js — thumbprint compareJavaScript
// x5t#S256 == base64url( SHA-256( DER-encoded client certificate ) )
var digestHex = KJUR.crypto.Util.hashHex(certHex, 'sha256');
var computed  = hextob64u(digestHex);        // base64url, no padding
var match     = (computed === expected);     // expected = cnf['x5t#S256']
context.setVariable('cert.binding.match', match);

Robustness details from the script:

  • Fail closed. cert.binding.match defaults to false; a match must be positively proven.
  • Certificate source fallback. Reads, in order, request.header.x-forwarded-client-cert, the native tls.client.raw.cert, then request.header.x-client-cert — covering Apigee X behind a Google Cloud HTTPS LB (which forwards the cert as a header) and hybrid topologies.
  • Format tolerance. Handles XFCC/Envoy Cert="...", URL-encoded PEM, literal-newline PEM, and bare base64 DER, normalising each to DER hex (pemtohex / b64utohex).
  • Diagnostics land in cert.binding.computed, cert.binding.expected, and cert.binding.error.
FAPI relevance & threat

This is the single biggest mitigation against token theft. Even if an access token leaks (log, referrer, breached store), it is worthless to an attacker who cannot also present the client certificate's private key on the TLS connection. It closes the "exported bearer token" gap that plain OAuth2 leaves open.

Configuration knobs

  • The forwarded-cert header name — align x-forwarded-client-cert / x-client-cert with how your load balancer presents the client certificate.
  • Sequence — keep this flow after sf-jwt-validation so jwt.VJWT-AccessToken.claim.cnf is populated.

5 · The x-fapi-* headers

Enforced in sf-fapi-headers

Guarantees the traceability contract FAPI requires: a well-formed correlation id on every request and response, plus validated end-user context — with no JavaScript, using only policy conditions and regexes.

NameTypePurpose
RF-InvalidInteractionIdRaiseFault400 when a present x-fapi-interaction-id is not an RFC 4122 UUID.
AM-InteractionIdAssignMessageEcho the caller's id or mint {messageid}; publish fapi.interaction.id and re-set the request header.
RF-InvalidCustomerIpRaiseFault400 when a present x-fapi-customer-ip-address is not a valid IPv4 or IPv6 address.

Step order & conditions

  • RF-InvalidInteractionId — when the header is present and fails the UUID regex (8-4-4-4-12 hex). Absent is allowed.
  • AM-InteractionId — always; Ref the caller header, Value falls back to {messageid}, then writes fapi.interaction.id and the request header.
  • RF-InvalidCustomerIp — when the header is present and fails the IPv4/IPv6 regex.
policies/AM-InteractionId.xmlXML
<AssignVariable>
  <Name>fapi.interaction.id</Name>
  <Ref>request.header.x-fapi-interaction-id</Ref>   <!-- echo caller's id -->
  <Value>{messageid}</Value>                       <!-- else mint a unique id -->
</AssignVariable>
FAPI relevance & threat

A single, validated fapi.interaction.id threads through every log line and every fault envelope, making cross-system incident forensics possible. Strict UUID / IP validation is also input hygiene: it rejects header values crafted to inject content into logs or downstream systems, and the value is reused (not re-derived) so faults always correlate.

Configuration knobs

  • The UUID and IPv4/IPv6 regexes in default.xml — already strict; adjust only if your profile differs.
  • Extend with other OB headers (e.g. x-fapi-auth-date, x-fapi-financial-id) following the same present-and-invalid pattern.

Production hardening checklist

Repoint these placeholders before any real deployment

Every cryptographic anchor below is a non-production placeholder.

  • JWKS / issuer / audience — auth.example.com URIs, Issuer, and Audience in VJWT-AccessToken; the signing JWKS in VJWS-Request.
  • Algorithm — promote VJWT-AccessToken from RS256 to PS256 to match the FAPI-advanced preference (JWS and JWE are already on the FAPI algorithms).
  • JWE keys — supply private.jwe.key / public.jwe.key from an encrypted KVM; remove the REPLACE_WITH_KVM_VALUE_* fallbacks' reliance.
  • Enforce, don't skip — make x-jws-signature mandatory on write/payment proxies.
  • Cert header — confirm the LB's forwarded-client-cert header name matches the fallbacks in certBinding.js.
