Shared flow

sf-security-headers

sf-security-headers is the response-hardening flow. It runs once, in the response path, and does three jobs with a single AssignMessage policy (AM-SecurityHeaders): it sets a fixed set of strict security headers (HSTS, X-Content-Type-Options, X-Frame-Options, a deny-all CSP, Cache-Control: no-store, Pragma, Referrer-Policy), it echoes the FAPI correlation id resolved earlier in the request flow back onto x-fapi-interaction-id, and it strips the Server and X-Powered-By fingerprint headers. It is pure policy — no JavaScript — and every value is a static constant except the echoed interaction id.

Where it sits

This is the last thing that touches the response before it leaves the gateway. It runs in the response/PostFlow so it can harden whatever the target returned — and, when attached to fault rules, whatever an error flow produced — without depending on any upstream behaviour.

The bundle is deliberately minimal: one policy, one default sub-flow, one step.

NameTypePurpose
AM-SecurityHeaders AssignMessage Targets the response message. Sets eight response headers (seven static security headers plus the echoed x-fapi-interaction-id) and removes the Server and X-Powered-By headers. continueOnError="false", IgnoreUnresolvedVariables="true".

The whole sub-flow is a single unconditional step — there are no conditions and no branching:

sharedflows/default.xmlXML
<SharedFlow name="default">
  <Step>
    <Name>AM-SecurityHeaders</Name>
  </Step>
</SharedFlow>

What the response carries afterward

Every header below is set verbatim from AM-SecurityHeaders. The values are hard-coded constants — only x-fapi-interaction-id is dynamic.

HeaderValueWhy
Strict-Transport-Security max-age=31536000; includeSubDomains Forces HTTPS for one year (31 536 000 s) across all sub-domains. No preload token, so the host is not committing to the browser preload list.
X-Content-Type-Options nosniff Stops browsers from MIME-sniffing a response away from its declared Content-Type — relevant if any error page or document is ever returned.
X-Frame-Options DENY Forbids the response being framed anywhere, blocking clickjacking. Belt-and-braces with the CSP below.
Content-Security-Policy default-src 'none' The strictest possible policy: a deny-all default with no allowed sources. Correct for a pure JSON API that should never load scripts, styles, frames, or any sub-resource.
Cache-Control no-store Prohibits any caching of the response — financial data must not persist in browser, proxy, or CDN caches.
Pragma no-cache HTTP/1.0 backstop for the no-store directive, for legacy intermediaries that ignore Cache-Control.
Referrer-Policy no-referrer Ensures no Referer header (which could leak resource ids or query data) is sent on any onward navigation.
x-fapi-interaction-id {fapi.interaction.id} Plays the FAPI correlation id back to the caller. The variable is resolved earlier in the request flow by sf-fapi-headers.

The three responsibilities

Although it is one policy, AM-SecurityHeaders does three distinct things. Read top to bottom they are:

Set the static hardening headers

The seven constant headers above are written into the response. Because AssignTo uses createNew="false" with type="response", the policy mutates the existing response in place — the target's status line and body are preserved, only headers change.

Echo the FAPI interaction id

The eighth Set header resolves {fapi.interaction.id} — the correlation id minted or echoed by sf-fapi-headers on the way in — and writes it onto the response x-fapi-interaction-id header, satisfying the FAPI play-back requirement.

Strip the fingerprint headers

The Remove block deletes Server and X-Powered-By, so the response does not advertise the gateway or backend technology stack to an attacker doing reconnaissance.

The policy

This is the entire flow — there is nothing else to read. Note the single Set block holding all eight headers and the Remove block listing the two stripped headers:

policies/AM-SecurityHeaders.xmlXML
<AssignMessage continueOnError="false" enabled="true" name="AM-SecurityHeaders">
  <AssignTo createNew="false" type="response"/>
  <IgnoreUnresolvedVariables>true</IgnoreUnresolvedVariables>
  <Set>
    <Headers>
      <Header name="Strict-Transport-Security">max-age=31536000; includeSubDomains</Header>
      <Header name="X-Content-Type-Options">nosniff</Header>
      <Header name="X-Frame-Options">DENY</Header>
      <Header name="Content-Security-Policy">default-src 'none'</Header>
      <Header name="Cache-Control">no-store</Header>
      <Header name="Pragma">no-cache</Header>
      <Header name="Referrer-Policy">no-referrer</Header>
      <!-- Echo the FAPI interaction id resolved earlier in the request flow. -->
      <Header name="x-fapi-interaction-id">{fapi.interaction.id}</Header>
    </Headers>
  </Set>
  <Remove>
    <Headers>
      <Header name="Server"/>
      <Header name="X-Powered-By"/>
    </Headers>
  </Remove>
</AssignMessage>
Why IgnoreUnresolvedVariables="true" matters here

The only dynamic value is {fapi.interaction.id}. If that variable were ever unset, a false setting would fault the whole response. With it true, an unresolved id degrades gracefully (the header resolves to empty) rather than turning a successful business response into a 500. In normal operation sf-fapi-headers guarantees the variable is present, so this is purely a safety net.

FAPI relevance

FAPI 1.0 Advanced and the UK Open Banking standard push security decisions to the edge and forbid leaking data through transport-layer side channels. This flow encodes several of those obligations:

  • Interaction-id play-back. The profile requires the ASPSP to return the x-fapi-interaction-id on the response. This flow is where that promise is kept for the response path, using the same fapi.interaction.id that sf-error-handling and sf-logging rely on.
  • No caching of financial data. Cache-Control: no-store plus the Pragma backstop ensure account and transaction responses are never written to any intermediate cache.
  • Transport hardening. HSTS keeps the channel on TLS, which underpins the mTLS sender-constraining done by sf-mtls-binding.
  • Reduced attack surface and reconnaissance. The deny-all CSP, frame denial, MIME-sniff protection, no-referrer policy, and stripped Server / X-Powered-By headers collectively starve an attacker of both client-side execution vectors and stack-fingerprinting hints.
Attach it to fault paths too, or errors stay un-hardened

The bundle only sets headers; it does not decide where it runs. To guarantee that error responses also carry the security headers and the echoed interaction id, wire sf-security-headers into the proxy response PostFlow and reference it from your FaultRules / DefaultFaultRule as well — otherwise a response produced by an error flow can bypass it.

Configuration knobs for production

The values ship as secure defaults. The items below are the ones a production team should consciously confirm or tighten for its own estate.

KnobWhereDefault & production guidance
HSTS preload Strict-Transport-Security value Default omits preload. Add it (and submit the host to the preload list) only once you are certain every sub-domain will serve HTTPS indefinitely — it is hard to reverse.
HSTS max-age Strict-Transport-Security value 31536000 (1 year) is the recommended floor. Keep at or above this; lower values weaken the guarantee.
CSP scope Content-Security-Policy value default-src 'none' is correct for a JSON-only API. Loosen only if a specific endpoint must return browser-rendered content; never relax it API-wide.
Header coverage on faults proxy PostFlow & FaultRules Reference this flow from both the response PostFlow and the fault rules so success and error responses are hardened identically (see warning above).
Extra stripped headers Remove block Strips Server and X-Powered-By. Add any other backend leak headers your targets emit (e.g. X-AspNet-Version, framework-specific banners).

Contract: what the caller receives

Always true after this flow

The response carries all seven static hardening headers and an x-fapi-interaction-id matching the request's correlation id, and it no longer advertises Server or X-Powered-By.

Never happens

The flow never inspects or rewrites the response body, never changes the status code, and never branches on a condition — it is a deterministic, body-agnostic header pass.
