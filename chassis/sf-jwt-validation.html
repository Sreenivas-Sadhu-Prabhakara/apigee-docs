Shared flow · OAuth2 / FAPI

sf-jwt-validation

sf-jwt-validation proves the caller is carrying a valid, in-scope OAuth2 access token before any request reaches the accounts backend. It runs four policies in order: extract the bearer token from the Authorization header, fault 401 if no token was presented, VerifyJWT the access token's signature/issuer/audience/expiry against the authorization server's JWKS, then fault 403 if the verified token does not carry an accounts scope. There is no token-introspection round trip — verification is stateless and offline against rotating JWKS keys.

This flow is the authorization gate of the FAPI 1.0 Advanced chassis. Network-edge concerns (mTLS, headers, preflight) run before it; payload-integrity concerns (JWS, JWE) run after it. Its single job is: is this a genuine, unexpired, correctly-audienced token, and does it grant the accounts permission?

Policies in this bundle

Four policies, declared in sf-jwt-validation.xml and sequenced by sharedflows/default.xml.

NameTypePurpose
EV-BearerToken ExtractVariables Parse the Authorization header with pattern Bearer {oauth.access.token}, landing the raw JWT in the oauth.access.token variable.
RF-MissingToken RaiseFault Return 401 UK.OBIE.Unauthorized when no token (or an empty token) was presented.
VJWT-AccessToken VerifyJWT Verify the access-token JWT — RS256 signature via JWKS, plus issuer, audience and expiry checks.
RF-InsufficientScope RaiseFault Return 403 UK.OBIE.Forbidden when the verified token's scope claim does not contain accounts.

Step order & conditions

The sequence below is exactly what sharedflows/default.xml executes. Two of the four steps are conditional; the extract and verify steps always run.

EV-BearerToken — always runs

ExtractVariables reads the request Authorization header and applies the case-insensitive pattern Bearer {oauth.access.token}. The brace name is the target variable, so the token lands in exactly oauth.access.token with no variable prefix. IgnoreUnresolvedVariables is true — a missing header does not fault here; the next step owns that decision.

RF-MissingToken — condition

Runs only when (oauth.access.token = null) or (oauth.access.token = ""). Raises 401 Unauthorized with a WWW-Authenticate: Bearer error="invalid_token" header and an OBIE error body. This terminates the request before VerifyJWT.

VJWT-AccessToken — always runs (when reached)

VerifyJWT validates the JWT in oauth.access.token: signature against the JWKS, iss against the configured issuer, aud against the configured audience, and exp for expiry. A failure raises Apigee's built-in JWT fault, surfaced as an error by sf-error-handling. On success it publishes claim variables such as jwt.VJWT-AccessToken.claim.scope.

RF-InsufficientScope — condition

Runs only when NOT(jwt.VJWT-AccessToken.claim.scope ~ "*accounts*"). Raises 403 Forbidden with WWW-Authenticate: Bearer error="insufficient_scope" and an OBIE error body. A token that is valid but lacks the accounts permission is rejected here.

Why VerifyJWT runs unconditionally

VJWT-AccessToken carries no <Condition>. It is safe because RF-MissingToken has already terminated any request where oauth.access.token is null or empty. If a token is present, it must be verified — there is no path that skips verification while still presenting a token.

The policies, as written

EV-BearerToken — extract the bearer token

A pattern-match extract. The deliberate absence of <VariablePrefix> is what keeps the variable name clean enough for the downstream condition and the VerifyJWT <Source> to reference it directly.

policies/EV-BearerToken.xmlXML
<ExtractVariables name="EV-BearerToken">
  <Source clearPayload="false">request</Source>
  <Header name="Authorization">
    <Pattern ignoreCase="true">Bearer {oauth.access.token}</Pattern>
  </Header>
  <!-- No VariablePrefix: the brace name IS the full variable name. -->
  <IgnoreUnresolvedVariables>true</IgnoreUnresolvedVariables>
</ExtractVariables>

VJWT-AccessToken — verify the access token

Stateless verification against the authorization server's published keys. The JWKS URI is used (not an inline key) so signing keys can rotate without redeploying the proxy.

policies/VJWT-AccessToken.xmlXML
<VerifyJWT name="VJWT-AccessToken">
  <Algorithm>RS256</Algorithm>
  <Source>oauth.access.token</Source>
  <PublicKey>
    <JWKS uri="https://auth.example.com/.well-known/jwks.json"/>
  </PublicKey>
  <Issuer>https://auth.example.com</Issuer>
  <Audience>accounts-api</Audience>
</VerifyJWT>

RF-MissingToken & RF-InsufficientScope — the rejections

Both faults emit JSON in UK Open Banking error shape, set an RFC 6750 WWW-Authenticate header, and echo the FAPI interaction id so the failure is traceable end to end.

policies/RF-MissingToken.xml — FaultResponseXML
<Header name="WWW-Authenticate">Bearer error="invalid_token",
        error_description="Missing access token"</Header>
<Header name="x-fapi-interaction-id">{request.header.x-fapi-interaction-id}</Header>
<Payload contentType="application/json">{
  "Code": "401",
  "Id": "{request.header.x-fapi-interaction-id}",
  "Message": "Unauthorized",
  "Errors": [
    { "ErrorCode": "UK.OBIE.Unauthorized",
      "Message": "Missing or empty access token" }
  ]
}</Payload>
<StatusCode>401</StatusCode>

RF-InsufficientScope is structurally identical with error="insufficient_scope", UK.OBIE.Forbidden, and <StatusCode>403</StatusCode>.

FAPI relevance

This flow implements the resource-server side of FAPI 1.0 Advanced token handling.

Bearer extraction Stateless JWT verification JWKS key rotation Scope enforcement OBIE error contract
  • Signed access tokens. FAPI requires asymmetric signatures; this flow verifies RS256 (with PS256 a one-line change — see below). No HMAC, no shared secret.
  • Issuer & audience binding. <Issuer> and <Audience> ensure a token minted for a different AS or a different API is rejected, blocking token-substitution attacks.
  • Correct HTTP semantics. Missing credential → 401 with WWW-Authenticate: invalid_token; valid-but-under-privileged → 403 with insufficient_scope, per RFC 6750.
  • Traceability. Every fault echoes x-fapi-interaction-id, satisfying the FAPI interaction-tracing requirement even on rejected calls.
Scope check is a substring match

The condition scope ~ "*accounts*" is a wildcard match against the whole space-delimited scope claim. It accepts any token whose scope string contains the substring accounts (e.g. accounts, accounts:read, openid accounts). For finer-grained authorization (read vs. write, consent-scoped grants), tighten this condition or add a dedicated scope policy downstream.

Configuration knobs

These are the placeholder values to change before production. All live in VJWT-AccessToken.xml except the scope substring, which is in sharedflows/default.xml.

Algorithm

<Algorithm>RS256</Algorithm> is the shipped default. FAPI prefers PS256 (RSASSA-PSS); to enforce it, change the element to <Algorithm>PS256</Algorithm> and ensure the AS signs with the matching algorithm.

JWKS URI

https://auth.example.com/.well-known/jwks.json is a placeholder. Point it at your real authorization server's JWKS endpoint. For offline tests, swap the <JWKS/> for <PublicKey><Value>PEM</Value></PublicKey>.

Issuer

<Issuer>https://auth.example.com</Issuer> must exactly match the iss claim your AS stamps into access tokens.

Audience

<Audience>accounts-api</Audience> must match the aud claim expected by this resource server.

Required scope

The substring in NOT(...scope ~ "*accounts*") in default.xml. Change accounts to the scope your API requires, and consider an exact-match condition for stricter checks.

Error contract

The UK.OBIE.* error codes and messages in both RaiseFault payloads. Keep them aligned with your tenant's published OBIE error catalogue.

Do not ship the example issuer/audience

The *.example.com issuer and JWKS URI and the accounts-api audience are non-resolvable placeholders. Left unchanged, every VerifyJWT will fail (JWKS fetch error or issuer mismatch) and no request will reach the backend. Replace all four VerifyJWT values as the first step of any real deployment.

Where this fits

In the reference proxy this shared flow is invoked from the request PreFlow after the network and header flows and before payload-integrity validation. The cumulative order is:

sf-mtls-binding sf-preflight sf-fapi-headers sf-jwt-validation sf-jws-validation sf-jwe

By the time sf-jwt-validation completes, the request is known to carry a cryptographically valid, correctly-audienced, in-scope access token — the precondition every downstream policy assumes.
