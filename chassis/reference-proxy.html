Proxy & operations

Reference proxy

bank-accounts-v1 is the single API proxy that ties the whole chassis together. It exposes the UK Open Banking Account Information API at base path /open-banking/v1.1/accounts, runs the entire FAPI security chain as seven ordered FlowCallout steps in the request PreFlow, then — instead of calling a backend — serves a hard-coded OB Account response from one AssignMessage policy (AM-MockAccounts) in the GetAccounts flow. The PostFlow hardens and logs the response, an empty <RouteRule name="no-target"/> makes it a target-less mock, and a DefaultFaultRule routes every fault through FC-ErrorHandling so errors come back in OB format. It is one ProxyEndpoint (default), eleven policies, no JavaScript, and no TargetEndpoints.

Where it sits

This proxy is the consumer-facing edge. Everything documented under Shared flows is invoked from here; the proxy itself contributes only the routing surface, the flow that answers GET /accounts, and the wiring that decides which shared flow runs when.

The bundle manifest declares one endpoint and the eleven policies it references — the ten FlowCallout bridges into the shared flows plus the single AssignMessage mock builder:

apiproxy/bank-accounts-v1.xmlXML
<APIProxy name="bank-accounts-v1">
  <ConfigurationVersion majorVersion="4" minorVersion="0"/>
  <Basepaths>/open-banking/v1.1/accounts</Basepaths>
  <ProxyEndpoints>
    <ProxyEndpoint>default</ProxyEndpoint>
  </ProxyEndpoints>
  <Resources/>
  <TargetEndpoints/>   <!-- deliberately empty: this is a mock -->
</APIProxy>
The effective resource URL is /open-banking/v1.1/accounts/accounts

The base path is /open-banking/v1.1/accounts and the GetAccounts flow matches the path suffix /accounts, so the resource actually answers at /open-banking/v1.1/accounts/accounts. The mock's Links.Self value reflects exactly this. It is intentional in the sample and is the first thing to rename when you map this onto a real OB resource model.

Request lifecycle, end to end

A request flows through the default ProxyEndpoint in this order. The PreFlow runs unconditionally for every request; the conditional steps inside it are skipped when their guard is false; the matched flow then builds the body; the PostFlow always runs on the way out; and any fault anywhere short-circuits to the DefaultFaultRule.

PreFlow — Request

Seven security steps run in fixed order (see the table below). Four are unconditional; three are guarded by conditions on the inbound headers and token claims. Any step that rejects the request raises a fault and jumps to the fault rule.

Conditional flow match — GetAccounts

If proxy.pathsuffix MatchesPath "/accounts" and the verb is GET, the GetAccounts flow runs and its single response step AM-MockAccounts builds the OB Account body. No request-side steps exist on this flow.

RouteRule — no-target

The empty <RouteRule name="no-target"/> resolves with no TargetEndpoint, so Apigee never makes a backend call. The response already built by AM-MockAccounts is what continues outward.

PostFlow — Response

FC-SecurityHeaders then FC-Logging run on the response, in that order, hardening headers and emitting the structured audit log before the response leaves the gateway.

On any fault — DefaultFaultRule

A raised fault at any point above transfers control to <DefaultFaultRule name="default-fault">, whose single step FC-ErrorHandling normalises the error into the OB error envelope.

The PreFlow security chain

This is the heart of the proxy: the order in which the shared flows are applied, and the conditions that decide whether the optional ones run at all. Steps execute top to bottom; a step with no condition always runs.

StepConditionPurpose
FC-Preflight always Invokes sf-preflight: CORS preflight handling, basepath/verb sanity, request-size guards, early rejection of malformed requests.
FC-FapiHeaders always Invokes sf-fapi-headers: validates/echoes x-fapi-interaction-id (UUID), x-fapi-auth-date, x-fapi-customer-ip-address, x-fapi-financial-id.
FC-JwtValidation always Invokes sf-jwt-validation: verifies the access-token JWT against the JWKS and checks issuer, audience, expiry and scope.
FC-JwsValidation request.header.x-jws-signature != null Invokes sf-jws-validation only when a detached JWS is present: validates the PS256 / b64=false signature and OB critical headers.
FC-MtlsBinding jwt.VJWT-AccessToken.claim.cnf != null or request.header.x-client-cert != null Invokes sf-mtls-binding when the token is sender-constrained or a client cert was presented: matches the cnf.x5t#S256 claim to the cert thumbprint.
FC-Jwe content-type ~ "application/jose*" or ~ "application/jwe*" Invokes sf-jwe only for encrypted payloads: decrypts the JWE request body and exposes the cleartext downstream.
FC-TrafficManagement always Invokes sf-traffic-management: quota, spike arrest and concurrent-rate-limit controls keyed by client identity.
Order is a security decision, not a convenience

Identity is established before message integrity, binding and traffic shaping: FC-JwtValidation populates jwt.VJWT-AccessToken.claim.cnf, which the next step's FC-MtlsBinding condition reads. Re-ordering these steps would break the mTLS-binding guard (the cnf claim would not yet exist) and let unauthenticated traffic reach the quota counters. Treat the PreFlow sequence as load-bearing.

The PreFlow exactly as declared — note the three <Condition> guards and the empty <Response/> (all security work happens on the request side):

apiproxy/proxies/default.xmlXML
<PreFlow name="PreFlow">
  <Request>
    <Step><Name>FC-Preflight</Name></Step>
    <Step><Name>FC-FapiHeaders</Name></Step>
    <Step><Name>FC-JwtValidation</Name></Step>
    <Step>
      <Name>FC-JwsValidation</Name>
      <Condition>(request.header.x-jws-signature != null)</Condition>
    </Step>
    <Step>
      <Name>FC-MtlsBinding</Name>
      <Condition>(jwt.VJWT-AccessToken.claim.cnf != null) or (request.header.x-client-cert != null)</Condition>
    </Step>
    <Step>
      <Name>FC-Jwe</Name>
      <Condition>(request.header.content-type ~ "application/jose*") or (request.header.content-type ~ "application/jwe*")</Condition>
    </Step>
    <Step><Name>FC-TrafficManagement</Name></Step>
  </Request>
  <Response/>
</PreFlow>

The GetAccounts flow

One conditional flow answers the only modelled resource. It does nothing on the request side; on the response side it runs a single policy that synthesises the body.

The flow condition combines an OB-style path match with a verb check, so only GET on the /accounts suffix is served — anything else falls through to no matching flow:

apiproxy/proxies/default.xml — FlowsXML
<Flow name="GetAccounts">
  <Description>GET /accounts - returns the mock OB Account list.</Description>
  <Condition>(proxy.pathsuffix MatchesPath "/accounts") and (request.verb = "GET")</Condition>
  <Request/>
  <Response>
    <Step><Name>AM-MockAccounts</Name></Step>
  </Response>
</Flow>
NameTypePurpose
AM-MockAccounts AssignMessage Builds the mock OB Account Information response in place. AssignTo createNew="false" type="response", IgnoreUnresolvedVariables=true. Sets status 200, Content-Type: application/json, and a static OB Account JSON payload. There is no backend, so this policy is the response.

The PostFlow

The PostFlow has no request-side steps. On the response it runs two FlowCallouts in order, so every response — mock success today, and any real backend response later — is hardened and then logged:

StepTypePurpose
FC-SecurityHeaders FlowCallout Invokes sf-security-headers: sets HSTS, X-Content-Type-Options, CSP, Cache-Control: no-store and the echoed x-fapi-interaction-id; strips fingerprint headers.
FC-Logging FlowCallout Invokes sf-logging: emits structured audit/access logs correlated by x-fapi-interaction-id, with PII and secrets masked.

The no-target RouteRule — the mock pattern

An Apigee proxy normally ends with a RouteRule that names a TargetEndpoint. This proxy deliberately omits the target, which is what turns it into a self-contained mock.

apiproxy/proxies/default.xml — routingXML
<HTTPProxyConnection>
  <BasePath>/open-banking/v1.1/accounts</BasePath>
</HTTPProxyConnection>
<RouteRule name="no-target"/>   <!-- no <TargetEndpoint> -->

Because the RouteRule has an empty body, it matches unconditionally and routes to nothing. Apigee skips the entire TargetEndpoint round-trip and proceeds straight to the response PostFlow, carrying whatever the flow built. This is the canonical Apigee way to ship a contract-first mock: the proxy is fully wired and deployable, but no upstream system needs to exist. The manifest reinforces this with an empty <TargetEndpoints/> element.

From mock to real with two edits

To front a live ASPSP backend, add a TargetEndpoint (with its own mTLS and target SSL config), give the RouteRule a <TargetEndpoint> child, and remove the AM-MockAccounts step from the GetAccounts flow (or move it behind a condition). The entire PreFlow security chain, PostFlow and fault handling continue to apply unchanged — that separation is the point of the chassis.

The DefaultFaultRule

The endpoint declares one fault rule. Any fault raised by any PreFlow step, the flow policy, or the PostFlow transfers control here, and its single step normalises the error:

apiproxy/proxies/default.xml — faultsXML
<DefaultFaultRule name="default-fault">
  <Step><Name>FC-ErrorHandling</Name></Step>
</DefaultFaultRule>

FC-ErrorHandling invokes sf-error-handling, which renders the OB Read/Write error envelope { Code, Id, Message, Errors[] }, using the x-fapi-interaction-id as the error Id and a generic Message so internal detail never leaks to the caller.

Security headers may not run on the error path

The DefaultFaultRule runs only FC-ErrorHandling. Whether the PostFlow's FC-SecurityHeaders also executes on a faulted response depends on where the fault was raised. For production, reference sf-security-headers from the fault rule too, so error responses are hardened identically to success responses (see the sf-security-headers note).

FAPI relevance

The proxy contributes the composition that makes the chassis FAPI 1.0 Advanced compliant — the individual obligations live in the shared flows, but this file is where they are ordered and gated:

  • Defence in depth, in the right order. Identity (FC-JwtValidation) precedes message integrity (FC-JwsValidation), sender-constraining (FC-MtlsBinding) and rate controls — so each later check can rely on the claims and state the earlier ones established.
  • Conditional cryptography only when present. Detached-JWS validation, mTLS binding and JWE decryption are gated on the actual request shape, so the proxy enforces them when they apply without rejecting compliant requests that legitimately omit them.
  • Correlation everywhere. FC-FapiHeaders establishes the x-fapi-interaction-id early; the PostFlow echoes it and logs by it; the fault rule uses it as the error Id — one id threads request, response, log and error.
  • OB-shaped contracts. Both the success body (OB Account) and the error body (OB error envelope) match the UK Open Banking schemas a TPP expects.

Configuration knobs for production

The proxy ships as a working mock. The items below are what a team should consciously change when promoting it to front a real backend or a different resource.

KnobWhereDefault & production guidance
Base path / version HTTPProxyConnection > BasePath and <Basepaths> /open-banking/v1.1/accounts. Align with your OB API version and host routing; keep manifest and endpoint in sync.
Resource path GetAccounts flow Condition Matches suffix /accounts, producing the doubled /accounts/accounts URL. Rename to the real OB resource path and add flows for /accounts/{AccountId}, transactions, balances, etc.
Backend target RouteRule & TargetEndpoints Empty no-target RouteRule = mock. Add a TargetEndpoint with target mTLS and give the RouteRule a <TargetEndpoint> child to call a live ASPSP.
Mock vs. live response AM-MockAccounts step Static ACCT-0001 / Account Holder sample data. Remove or condition the step once a real target is wired; never ship the mock to production unguarded.
Fault-path hardening DefaultFaultRule Runs only FC-ErrorHandling. Add FC-SecurityHeaders (and optionally FC-Logging) so error responses are hardened and audited like successes.
Self link host AM-MockAccounts payload Links.Self is hard-coded to https://apigee/.... Replace with your real host (ideally derived from request context) so the OB pagination/self links are usable.

Contract: what this proxy guarantees

Always true

Every request runs the full PreFlow security chain in fixed order before any business logic; GET /open-banking/v1.1/accounts/accounts returns the OB Account body with 200; every response passes through FC-SecurityHeaders and FC-Logging; every fault is normalised by FC-ErrorHandling.

Never happens

No backend call is ever made (the RouteRule is target-less); no JavaScript runs in the proxy itself; the optional crypto steps never run unless their request shape is present; and no flow other than GetAccounts is served.
