Shared flow · Payload confidentiality

sf-jwe

sf-jwe provides JOSE payload encryption for the chassis: it decrypts an inbound request body and can encrypt an outbound response body. Both directions are a single JavaScript policy that runs one shared script, jwe.js, on top of the bundled jsrsasign library (KJUR.jose.JWE). The crypto profile is fixed to the FAPI / UK Open Banking shape — RSA-OAEP-256 key management and A256GCM content encryption — and the decrypt call passes an algorithm allow-list to block downgrade. The flow is additive and self-gating: only JS-DecryptJWE is stepped, only when the request Content-Type is a JOSE media type, and the script itself re-checks that the body is a 5-segment compact JWE before touching it. Keys come from a KVM (private.jwe.key / public.jwe.key); the in-file PEMs are non-functional placeholders.

This flow is the message-confidentiality layer. mTLS protects data in transit hop-by-hop; sf-jwe protects the payload itself end-to-end, so a sensitive request or response body stays ciphertext through any intermediary. It assumes the request has already been authenticated and integrity-checked by the flows ahead of it.

Policies in this bundle

Two policies, both Javascript, declared in sf-jwe.xml. They are deliberately identical except for one property — jwe.mode — which selects the direction inside the shared script.

NameTypePurpose
JS-DecryptJWE Javascript Runs jwe.js with jwe.mode = "decrypt". Replaces a compact-JWE request.content with its plaintext. Stepped in default.xml under a content-type condition.
JS-EncryptJWE Javascript Runs jwe.js with jwe.mode = "encrypt". Replaces response.content with a compact JWE. Defined but intentionally not stepped — attach it on the response flow of a consuming proxy.

Both policies carry timeLimit="5000" and continueOnError="false", and both pull in two resources:

jsc://jwe.js jsc://jsrsasign-all-min.js

jsrsasign-all-min.js (~342 KB) is referenced as an <IncludeURL> so its globals (KJUR, KEYUTIL) are in scope when jwe.js — the <ResourceURL> — executes.

Step order & conditions

sharedflows/default.xml steps exactly one policy. The encrypt policy is shipped but left unstepped on purpose.

JS-DecryptJWE — conditional on JOSE media type

Stepped with the condition (request.header.content-type ~ "application/jose*") or (request.header.content-type ~ "application/jwe*"). A plain application/json request never enters this policy, so non-encrypted traffic is untouched and pays no crypto cost.

Inside jwe.js — second guard

Even when the condition matches, the script re-validates that request.content is a compact JWE (exactly five dot-separated segments). If not, it sets jwe.decrypted = false and returns without modifying the body. The media-type condition and the structural guard are deliberately belt-and-braces.

JS-EncryptJWE — not stepped here

Response encryption is opt-in per proxy. There is no <Step> for it in default.xml; a consuming proxy attaches JS-EncryptJWE on its response flow (typically conditioned on the client's Accept header) when JWE responses are required.

One script, two policies

There is no separate decrypt and encrypt source. jwe.js holds both code paths; the policy <Property name="jwe.mode"> chooses which one runs. This keeps the key-handling, algorithm allow-list, and error contract identical across directions.

How jwe.js works

The whole script is an IIFE in "use strict". It resolves direction, dispatches to one of two functions, and never lets an exception escape the policy.

Mode resolution

Apigee surfaces a policy <Property> through the read-only global properties object — not through context.getVariable. The script reads properties["jwe.mode"] first, accepts a flow variable jwe.mode as an override, then defaults to "decrypt" for safety.

resources/jsc/jwe.js — directionJS
// Reset any prior diagnostic so a clean run does not inherit a stale error.
context.setVariable("jwe.error", "");

var mode;
try {
  if (typeof properties !== "undefined" && properties && properties["jwe.mode"]) {
    mode = properties["jwe.mode"];
  }
} catch (ePropsIgnored) { /* fall through to flow variable */ }
if (!mode) { mode = context.getVariable("jwe.mode"); }
if (!mode) { mode = "decrypt"; }

Decrypt — inbound request

The structural guard isCompactJWE returns true only for a string of exactly five base64url segments (header.key.iv.ciphertext.tag). The recipient RSA private key is read from private.jwe.key, parsed with KEYUTIL.getKey (a top-level jsrsasign global, not under KJUR), and the decrypt is constrained to the FAPI algorithm allow-list.

resources/jsc/jwe.js — doDecrypt()JS
var payload = context.getVariable("request.content");
if (!isCompactJWE(payload)) {       // not a JWE -> leave request as-is
  context.setVariable("jwe.decrypted", false);
  return;
}
var privateKeyPem = context.getVariable("private.jwe.key");   // from KVM
var prvKey = KEYUTIL.getKey(privateKeyPem);

// Restrict accepted algorithms to the FAPI profile to avoid alg-downgrade.
var result = KJUR.jose.JWE.decrypt(payload, prvKey, {
  alg: ["RSA-OAEP-256"],
  enc: ["A256GCM"]
});
context.setVariable("request.content", result);
context.setVariable("jwe.decrypted", true);
The algorithm allow-list is the security control

Passing { alg: ["RSA-OAEP-256"], enc: ["A256GCM"] } to KJUR.jose.JWE.decrypt is what makes this FAPI-safe. Without it, jsrsasign would honour whatever alg/enc the attacker put in the protected header, opening the door to algorithm-downgrade attacks. Keep this list narrow; widen it only for a documented reason.

Encrypt — outbound response

The plaintext (response.content) is coerced to a string, the recipient public key is read from public.jwe.key, and a JOSE header is built with the fixed profile plus an optional kid from jwe.kid. On success the body is replaced and the response Content-Type is set to application/jose.

resources/jsc/jwe.js — doEncrypt()JS
var pubKey = KEYUTIL.getKey(context.getVariable("public.jwe.key"));
var kid = context.getVariable("jwe.kid");   // optional key id

var header = {
  alg: "RSA-OAEP-256",
  enc: "A256GCM",
  cty: "application/json"
};
if (kid) { header.kid = kid; }

var jwe = KJUR.jose.JWE.encrypt(pubKey, JSON.stringify(header), plaintext);

context.setVariable("response.content", jwe);
context.setVariable("response.header.Content-Type", "application/jose");
context.setVariable("jwe.encrypted", true);

Error handling — the policy never faults

The dispatch is wrapped in a single try/catch. Any failure — bad key, malformed token, rejected algorithm — is recorded in jwe.error and swallowed; the message body is left untouched. So although the policies declare continueOnError="false", in practice no exception ever propagates out, and the proxy must inspect jwe.error to react.

resources/jsc/jwe.js — dispatchJS
try {
  if (mode === "encrypt") { doEncrypt(); }
  else { doDecrypt(); }
} catch (e) {
  // Never propagate the raw exception; record it for the fault-handling flow.
  context.setVariable("jwe.error", (e && e.message) ? e.message : String(e));
}
Check jwe.error downstream

Because exceptions are caught internally, a decrypt failure produces a successful policy and an unchanged (still-ciphertext) request.content, not an Apigee JWT-style fault. The consuming proxy should add a step after sf-jwe that raises an OB error envelope when jwe.error is non-empty — otherwise an undecryptable body could be forwarded to the backend verbatim.

Variable contract

Everything jwe.js reads and writes, in one place.

NameTypePurpose
jwe.modein (property/var)Direction selector — "decrypt" or "encrypt"; defaults to decrypt.
private.jwe.keyin (KVM)Recipient RSA private key (PEM) for decrypt.
public.jwe.keyin (KVM)Recipient/client RSA public key (PEM) for encrypt.
jwe.kidin (optional)Key id stamped into the JOSE header on encrypt, if present.
request.contentin/outRead as ciphertext, replaced with plaintext on a successful decrypt.
response.contentin/outRead as plaintext, replaced with a compact JWE on encrypt.
jwe.decryptedouttrue if a JWE was decrypted, false if the body was not a JWE.
jwe.encryptedouttrue once a response has been encrypted.
jwe.erroroutReset to "" each run; set to the caught exception message on failure.

Key handling

Keys are never embedded. Both directions read a PEM from a flow variable that a preceding policy must populate from a KVM — preferably an encrypted KVM via a KeyValueMapOperations policy with mapIdentifier="jwe-keys". The PEMs hard-coded in jwe.js are loud placeholders that exist only so local smoke tests fail visibly.

resources/jsc/jwe.js — placeholder (do NOT ship)JS
var privateKeyPem = context.getVariable("private.jwe.key");
if (!privateKeyPem) {
  privateKeyPem =
    "-----BEGIN PRIVATE KEY-----\n" +
    "REPLACE_WITH_KVM_VALUE_private.jwe.key\n" +
    "-----END PRIVATE KEY-----";
}
Wire up the KVM before enabling JWE

If private.jwe.key / public.jwe.key are not set, KEYUTIL.getKey is handed the REPLACE_WITH_KVM_VALUE_* placeholder, throws, and the message passes through unchanged with jwe.error populated. Add a KeyValueMapOperations step that loads the keys into these variables before the JS policy, and store the private key in an encrypted KVM.

FAPI relevance

This flow delivers payload confidentiality, the JOSE complement to the chassis's transport (mTLS) and integrity (JWS) layers.

JWE compact serialization RSA-OAEP-256 A256GCM (AEAD) Algorithm allow-list End-to-end confidentiality
  • Approved crypto profile. RSA-OAEP-256 for key management and A256GCM (authenticated encryption) for content matches the JWE profile recommended for FAPI / UK Open Banking sensitive payloads.
  • Downgrade resistance. The explicit alg/enc allow-list on decrypt prevents an attacker from forcing a weaker or none-style algorithm via the protected header.
  • Confidentiality beyond the TLS hop. The body stays encrypted through load balancers, proxies and logs — only the holder of private.jwe.key can read it.
  • Correct media typing. Encrypt sets Content-Type: application/jose and a cty: application/json header so the recipient knows it is JOSE wrapping JSON.
  • Additive, not mandatory. Non-JOSE requests pass untouched, so JWE can be turned on per route without breaking plaintext clients during rollout.

Configuration knobs

The placeholder and profile values to change before production.

Key sources

private.jwe.key and public.jwe.key must be loaded from a KVM (suggested mapIdentifier="jwe-keys") by a KeyValueMapOperations step ahead of the JS policy. Never leave the in-file PEM placeholders.

Crypto profile

The alg/enc allow-list (decrypt) and the JOSE header (encrypt) live in jwe.js: RSA-OAEP-256 + A256GCM. Changing the profile means editing the script, not the policy XML.

Trigger condition

The media-type match in default.xml (application/jose* / application/jwe*) decides when decrypt runs. Tighten or broaden it to match your clients' Content-Type.

Response encryption

JS-EncryptJWE is unstepped. Attach it on a proxy's response flow (often gated on Accept: application/jose) and supply jwe.kid if the client needs to select a key.

Time limit

Both policies set timeLimit="5000" (ms). Raise it only if large-payload RSA-OAEP operations approach the budget under load.

Failure handling

jwe.error is the only failure signal. Add a downstream step that converts a non-empty jwe.error into the tenant's OB Read/Write error envelope.

Where this fits

In the reference proxy sf-jwe is the last of the request-side security flows — after authentication and integrity, the message is finally decrypted for the backend. The cumulative request order is:

sf-mtls-binding sf-preflight sf-fapi-headers sf-jwt-validation sf-jws-validation sf-jwe

By the time sf-jwe finishes the request, the body has been decrypted in place for the target. On the way out, an optionally-attached JS-EncryptJWE re-seals the response for the client — the symmetric bookend of this flow.
