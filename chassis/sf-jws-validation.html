Shared flow · FAPI message integrity

sf-jws-validation

sf-jws-validation proves the request body was signed by the calling TPP and arrived unaltered — FAPI message integrity. It runs a single policy, VJWS-Request (VerifyJWS), which verifies the detached JWS carried in the x-jws-signature header against the request body supplied as DetachedContent. The signature must be PS256 (RSASSA-PSS), the signing key is resolved from a JWKS by the kid in the JWS protected header, and the four UK Open Banking critical headers (b64, iat, iss, tan) are enforced. The step is conditional on the signature header being present, so the demo GET (no body, no signature) still flows; any signature that is presented is fully verified.

This flow is the payload-integrity gate of the FAPI 1.0 Advanced chassis. It runs after the authorization gate (sf-jwt-validation) and answers one question: was this exact request body signed by the holder of the registered signing key, with the Open Banking critical headers intact?

Policies in this bundle

One policy, declared in sf-jws-validation.xml and sequenced by sharedflows/default.xml.

NameTypePurpose
VJWS-Request VerifyJWS Verify the detached JWS in x-jws-signature (PS256, b64=false per RFC 7797) against the request body, resolving the public key from the signing JWKS and enforcing the OB critical headers.

Step order & conditions

One conditional step. The condition is exactly what sharedflows/default.xml declares.

VJWS-Request — condition

Runs only when (request.header.x-jws-signature != null). If the header is present, the detached signature is fully verified; if it is absent, the step is skipped so a bodyless demo GET passes through. On a verification failure, VerifyJWS raises Apigee's built-in JWS fault, which sf-error-handling surfaces as the error response.

Presence is not enforced here — by design

The step is gated on x-jws-signature != null purely so the read-only demo path (a GET with no body) works without a signature. This means an attacker cannot bypass verification by sending a bad signature — but they could omit the header entirely. Production write/payment endpoints MUST enforce presence and reject any request missing x-jws-signature, per the OB Read/Write spec. See Configuration knobs below for how.

What "detached JWS" means here

A normal JWS is header.payload.signature — three base64url segments. A detached JWS omits the middle segment, transmitting header..signature; the payload is sent separately (here, as the request body). The verifier reattaches the body before checking the signature, which is exactly what <DetachedContent>request.content</DetachedContent> does.

x-jws-signature

Carries the protected header and the signature only — eyJ...header...<empty>.<sig>. The empty middle is the detached payload.

request.content

The raw request body. Supplied as DetachedContent, it is the payload that the signature is verified against — byte for byte.

b64=false (RFC 7797) — why the body is signed verbatim

UK Open Banking signs with the unencoded-payload option: the b64 protected-header parameter is false, so the body is not base64url-encoded before signing. The body bytes on the wire are the signed bytes. b64 is therefore listed as a critical header — a verifier that does not understand it must reject the JWS rather than silently treat the payload as encoded.

The policy, as written

The whole flow is this one VerifyJWS. The algorithm is pinned, the source and detached content are wired to the request, the key comes from a JWKS, and the OB critical headers are declared so they are not rejected as unrecognised.

policies/VJWS-Request.xmlXML
<VerifyJWS name="VJWS-Request">
  <DisplayName>VJWS-Request</DisplayName>
  <Algorithm>PS256</Algorithm>
  <Source>request.header.x-jws-signature</Source>
  <DetachedContent>request.content</DetachedContent>
  <PublicKey>
    <JWKS uri="https://auth.example.com/.well-known/signing-jwks.json"/>
  </PublicKey>
  <CriticalHeaders>b64,http://openbanking.org.uk/iat,http://openbanking.org.uk/iss,http://openbanking.org.uk/tan</CriticalHeaders>
</VerifyJWS>

There is no inline <Id> for the key: the kid in the JWS protected header selects the matching key from the JWKS, so the TPP's signing keys can rotate without redeploying the proxy.

sharedflows/default.xmlXML
<SharedFlow name="default">
  <Step>
    <Name>VJWS-Request</Name>
    <Condition>(request.header.x-jws-signature != null)</Condition>
  </Step>
</SharedFlow>

The four critical headers

The crit list in the JWS protected header names parameters the signer considers mandatory. VerifyJWS must be told which of these it understands via <CriticalHeaders>; any unlisted crit entry causes the verification to fail. These four are the UK OB Read/Write set.

HeaderMeaning
b64RFC 7797 unencoded-payload flag — false for OB, signalling the body is signed verbatim.
http://openbanking.org.uk/iatIssued-at time of the signature (epoch seconds), bounding signature freshness.
http://openbanking.org.uk/issIssuer of the signature — the signing party's identifier.
http://openbanking.org.uk/tanTrust Anchor used to source the signing certificate (e.g. openbanking.org.uk).

FAPI relevance

This flow implements the resource-server side of FAPI 1.0 Advanced request-object message integrity.

Detached JWS PS256 only b64=false (RFC 7797) JWKS key rotation OB critical headers
  • Non-repudiation & integrity. The body is bound to the TPP's signing key; any in-transit modification or a forged body fails verification. mTLS protects the channel — this protects the message.
  • Algorithm pinning. <Algorithm>PS256</Algorithm> accepts only RSASSA-PSS. There is no none path and no HMAC, blocking algorithm-confusion downgrade attacks.
  • Critical-header enforcement. Declaring the OB crit set means a signature that omits or alters b64/iat/iss/tan is rejected rather than silently accepted.
  • Key agility. Verification is offline against the signing JWKS; the kid selects the key, so signer key rotation needs no proxy change.

Configuration knobs

These are the placeholder values to change before production. The policy itself lives in VJWS-Request.xml; the presence rule lives in sharedflows/default.xml.

Signing JWKS URI

https://auth.example.com/.well-known/signing-jwks.json is a non-resolvable placeholder. Point it at the TPP / OB signing key set for each environment — KVM-backed in production. Note this is the signing JWKS, distinct from the access-token JWKS in sf-jwt-validation.

Enforce signature presence

For write/payment proxies, drop the x-jws-signature != null condition (run VJWS-Request unconditionally) or add a RaiseFault that returns 400 UK.OBIE.Signature.Missing when the header is absent. Do not ship the demo's conditional gate on a payment endpoint.

Algorithm

<Algorithm>PS256</Algorithm> is correct for UK OB and should stay pinned. Do not widen it to a list; one algorithm per directory keeps downgrade attacks off the table.

Critical headers

The <CriticalHeaders> set must match the crit values your signers actually emit. If your tenant adds OB header parameters, list them here too — an unlisted crit entry fails verification.

Do not ship the example JWKS URI — or the conditional gate on writes

The auth.example.com signing JWKS is a placeholder; left unchanged, every presented signature fails the JWKS fetch. And because the step is conditioned on header presence, a missing signature is silently allowed — acceptable for the read-only demo, unacceptable for any payment or write endpoint. Before production: replace the JWKS URI and enforce signature presence.

Where this fits

In the reference proxy this shared flow is invoked from the request PreFlow after the authorization gate and before payload confidentiality (sf-jwe). The cumulative order is:

sf-mtls-binding sf-preflight sf-fapi-headers sf-jwt-validation sf-jws-validation sf-jwe

By the time sf-jws-validation completes, a signed request is known to carry an intact, PS256-signed body from the registered signing key with valid OB critical headers — the integrity precondition the backend relies on.
