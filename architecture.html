Start · The chassis pattern

Architecture

Apigee Docs is a policy chassis: API proxies stay thin and delegate every cross-cutting concern to ten versioned shared flows through FlowCallout policies wired into the proxy's PreFlow, PostFlow, and DefaultFaultRule. The reference proxy bank-accounts-v1 carries the entire FAPI 1.0 Advanced posture in ten one-line FlowCallouts and a single mock-response policy — it contains no security logic of its own. A new bank API onboards by copying that same FlowCallout chain and adding its resource flows; it writes essentially no security code. Because FlowCallouts bind to a shared flow by name, the bundles must already be deployed in the environment before the proxy — which is exactly the order tools/deploy.sh enforces.

The chassis pattern

Centralise the hard, regulated parts once; let every proxy inherit them.

In a FAPI deployment the security surface — token validation, detached message signing, sender-constrained binding, payload encryption, traffic governance, audit logging — is large, subtle, and identical across every endpoint a bank exposes. Re-implementing it per proxy guarantees drift and audit gaps. The chassis pattern inverts that: the security surface lives in a fleet of shared flows (Apigee's reusable policy bundles), and each proxy is reduced to two responsibilities.

What the proxy owns

Its BasePath, its resource Flows (verb + path conditions), and what it does with a matched request — here, the AM-MockAccounts AssignMessage that returns a mock OB Account list. Business shape only.

What the chassis owns

Preflight guards, x-fapi-* handling, JWT / JWS / JWE / mTLS validation, quota and spike arrest, response hardening, OB-format faults, and structured logging — the entire FAPI 1.0 Advanced profile, versioned independently of any proxy.

Why "chassis"

As stated in the repository README: a team shipping a new API "inherits the entire FAPI security posture by wiring in FlowCallouts; they write almost no security code themselves." The proxy is the body panel; the shared flows are the frame, drivetrain, and brakes that every model reuses unchanged.

How a thin proxy delegates via FlowCallout

A FlowCallout is a policy whose only job is to invoke a shared-flow bundle by name and return. Each of the chassis's ten concerns is exposed to the proxy as one such policy. They are deliberately trivial — here is the preflight callout in full:

apiproxies/bank-accounts-v1/apiproxy/policies/FC-Preflight.xmlXML
<FlowCallout name="FC-Preflight">
  <DisplayName>FC-Preflight</DisplayName>
  <Description>Invokes sf-preflight: CORS preflight handling, basepath/verb
    sanity, request size guards, and early rejection of malformed requests.</Description>
  <SharedFlowBundle>sf-preflight</SharedFlowBundle>
</FlowCallout>

The single load-bearing line is <SharedFlowBundle>sf-preflight</SharedFlowBundle>. There is no revision, no environment, no URL — the binding is by bundle name, resolved at runtime against whatever revision of sf-preflight is deployed in the current environment. That late binding is what makes the chassis upgradeable: improve a shared flow, redeploy it, and every proxy that calls it picks up the change without being touched.

The proxy then arranges these callouts into Apigee's standard request/response lifecycle. The PreFlow runs the inbound security chain; a resource Flow produces the response; the PostFlow hardens and logs it; and a DefaultFaultRule normalises any error:

apiproxies/bank-accounts-v1/apiproxy/proxies/default.xmlXML
<!-- PreFlow / Request: the inbound FAPI security chain -->
<Step><Name>FC-Preflight</Name></Step>
<Step><Name>FC-FapiHeaders</Name></Step>
<Step><Name>FC-JwtValidation</Name></Step>
<Step>
  <Name>FC-JwsValidation</Name>
  <Condition>(request.header.x-jws-signature != null)</Condition>
</Step>
<Step>
  <Name>FC-MtlsBinding</Name>
  <Condition>(jwt.VJWT-AccessToken.claim.cnf != null) or (request.header.x-client-cert != null)</Condition>
</Step>
<Step>
  <Name>FC-Jwe</Name>
  <Condition>(request.header.content-type ~ "application/jose*") or (request.header.content-type ~ "application/jwe*")</Condition>
</Step>
<Step><Name>FC-TrafficManagement</Name></Step>

Diagram: proxy → the ten shared flows

One request through bank-accounts-v1 touches all ten shared flows across four phases. Three of the PreFlow callouts are conditional — they fire only when the relevant material is present — while the rest always run. Read top to bottom as the execution order of a request to GET /open-banking/v1.1/accounts/accounts.

FC-Preflight → sf-preflight

PreFlow · Requestalways

Spike arrest plus fail-fast guards on HTTP method (405), media type (415), and payload size > 1 MiB (413). The first line of defence; rejects malformed traffic before any other work.

FC-FapiHeaders → sf-fapi-headers

PreFlow · Requestalways

Validates and echoes the FAPI interaction headers — x-fapi-interaction-id (UUID), x-fapi-auth-date, x-fapi-customer-ip-address, x-fapi-financial-id.

FC-JwtValidation → sf-jwt-validation

PreFlow · Requestalways

Verifies the OAuth2 access token against the JWKS URI and checks issuer, audience, expiry, and scope. Populates jwt.VJWT-AccessToken.* for the steps that follow.

FC-JwsValidation → sf-jws-validation

PreFlow · Requestif x-jws-signature present

Validates the detached JWS in x-jws-signature (PS256, b64=false per RFC 7797) including the OB critical headers b64, iat, iss, tan.

FC-MtlsBinding → sf-mtls-binding

PreFlow · Requestif cnf claim or x-client-cert

Enforces sender-constrained tokens by comparing the token's cnf x5t#S256 claim with the base64url SHA-256 of the presented client mTLS certificate (DER).

FC-Jwe → sf-jwe

PreFlow · Requestif content-type jose / jwe

Decrypts JWE-encrypted request payloads (application/jose or application/jwe) and exposes the cleartext to downstream policies.

FC-TrafficManagement → sf-traffic-management

PreFlow · Requestalways

Applies quota, spike arrest, and concurrent-rate-limit controls keyed by client / app identity — the last PreFlow gate before the resource flow.

AM-MockAccounts

GetAccounts · Response

The proxy's only business policy. On (pathsuffix MatchesPath "/accounts") and (verb = "GET") it returns the mock OB Account list. The single RouteRule is no-target — there is no backend.

FC-SecurityHeaders → sf-security-headers

PostFlow · Responsealways

Sets response-hardening headers (HSTS, X-Content-Type-Options, Cache-Control, and more) and strips sensitive internal headers.

FC-Logging → sf-logging

PostFlow · Responsealways

Emits structured audit / access logs correlated by x-fapi-interaction-id while masking PII and secrets.

FC-ErrorHandling → sf-error-handling

DefaultFaultRuleon any fault

Catches faults raised anywhere above and normalises them into the OB Read/Write error format { Code, Id, Message, Errors[] } — Id is the x-fapi-interaction-id, with a generic Message and PII masking.

Repository layout

The repository separates the reusable chassis (shared-flows/) from the proxies that consume it (apiproxies/), with deploy tooling and keyless CI alongside.

apigee-bank-chassis/tree
apigee-bank-chassis/
├── shared-flows/            # The reusable chassis (10 shared flows)
│   ├── sf-preflight/                # spike arrest, method/content-type/size guards
│   ├── sf-fapi-headers/             # x-fapi-interaction-id / customer-ip handling
│   ├── sf-jwt-validation/           # OAuth2 access-token verification (JWKS)
│   ├── sf-jws-validation/           # detached JWS request-signature (PS256)
│   ├── sf-jwe/                      # JWE decrypt/encrypt (jsrsasign)
│   ├── sf-mtls-binding/             # cnf x5t#S256 certificate binding
│   ├── sf-security-headers/         # HSTS, CSP, nosniff, cache-control
│   ├── sf-traffic-management/       # quota + per-client spike arrest
│   ├── sf-error-handling/           # OB-format faults, PII masking
│   └── sf-logging/                  # structured Cloud Logging + analytics
├── apiproxies/
│   └── bank-accounts-v1/            # reference proxy (AssignMessage mock target)
├── config/                  # env config, KVMs, target servers
├── tools/                   # deploy.sh, vendored libs, test helpers
└── .github/workflows/       # WIF-based CI/CD (workflow_dispatch + v* tags)

Each shared flow is a self-contained bundle: shared-flows/<name>/sharedflowbundle/ holds the bundle descriptor (<name>.xml), the execution order (sharedflows/default.xml), and the policies it runs (policies/*.xml). The proxy bundle mirrors this shape: its apiproxy/bank-accounts-v1.xml descriptor lists all eleven policies (the ten FlowCallouts plus AM-MockAccounts) and the single default ProxyEndpoint.

Onboarding a new bank API

Copy the chain, declare your resources, ship. The security work is already done.

Because the FAPI posture lives entirely in shared flows, a new proxy reuses it by copying FlowCallout wiring — not by re-implementing policies. The steps below produce a fully FAPI-compliant proxy with zero new security code.

Scaffold the proxy bundle

Create apiproxies/<new-api>/apiproxy/ with a descriptor and a proxies/default.xml. Set the <BasePath> for the new API (for example /open-banking/v1.1/payments).

Copy the ten FlowCallout policies

Copy policies/FC-*.xml from bank-accounts-v1 verbatim. Each is a name-bound <SharedFlowBundle> reference, so they are identical across proxies and need no edits.

Reuse the PreFlow / PostFlow / fault chain

Copy the PreFlow Request steps (including the three conditions on FC-JwsValidation, FC-MtlsBinding, FC-Jwe), the PostFlow Response steps (FC-SecurityHeaders, FC-Logging), and the DefaultFaultRule (FC-ErrorHandling). This is the entire security inheritance.

Write only your business flows

Add resource <Flow>s with verb + path conditions and a target (or, as in the reference, an AssignMessage mock and a no-target RouteRule). This is the only code unique to your API.

Register policies in the descriptor

List the copied FlowCallouts plus your business policies in the proxy bundle's <Policies> block, then add the proxy name to the SHARED_FLOWS/proxy section of tools/deploy.sh (or deploy it directly).

Net new security code: none

Every regulated control — token, signature, binding, encryption, traffic, headers, faults, logging — is inherited by reference. A new team owns only its BasePath, its resource flows, and its target. The audit surface stays in one place.

Shared-flow versioning & the deploy dependency

FlowCallouts reference a bundle by name only (<SharedFlowBundle>sf-preflight</SharedFlowBundle>); they pin no revision. Apigee resolves the name at runtime to the revision currently deployed in the environment. Versioning is therefore environment-scoped:

  • Import creates a new numbered revision of the bundle.
  • Deploy makes one revision live in a given environment.
  • The proxy always calls whatever revision is live — so an upgrade is "redeploy the shared flow," and callers update with no change.

The deploy script uses apigeecli ... --ovr --wait to import-and-deploy each bundle, overwriting the live revision in one step. The bundle descriptors carry ConfigurationVersion majorVersion="4" minorVersion="0", the Apigee bundle format version (not a semantic version of the flow).

Deploy order is a hard dependency

A proxy import fails if a shared flow it references is not yet present in the target environment. Name-based binding means the bundles must exist before the proxy. tools/deploy.sh encodes exactly this: it deploys all ten shared flows first, then the proxy.

tools/deploy.shbash
# --- Deploy shared flows -----------------------------------------------------
for sf in "${SHARED_FLOWS[@]}"; do
  "$CLI" sharedflows create bundle \
    -n "$sf" -f "$ROOT/shared-flows/$sf/sharedflowbundle" \
    -e "$ENV" --ovr --wait \
    -o "$ORG" -t "$TOKEN" --disable-check --no-warnings
done

# --- Deploy the reference proxy (only after the flows exist) -----------------
"$CLI" apis create bundle \
  -n "$PROXY" -f "$ROOT/apiproxies/$PROXY/apiproxy" \
  -e "$ENV" --ovr --wait \
  -o "$ORG" -t "$TOKEN" --disable-check --no-warnings

Two bundles have an extra build-time dependency: before deploying, the script stages the vendored jsrsasign-all-min.js into sf-jwe and sf-mtls-binding (the JavaScript that performs JWE crypto and the x5t#S256 thumbprint). The SHARED_FLOWS array in the script lists all ten bundles, and a new proxy is added after them — preserving the flows-before-proxy invariant for every deploy.
