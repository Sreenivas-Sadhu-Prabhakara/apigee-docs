sf-fapi-headers

sf-fapi-headers validates and normalizes the two FAPI HTTP request headers that flow through every protected API: x-fapi-interaction-id and x-fapi-customer-ip-address. It does three things, in order — reject a present-but-malformed interaction id, then echo-or-generate the interaction id and publish it on the fapi.interaction.id context variable, then reject a present-but-malformed customer IP. The result is a guaranteed, downstream-readable correlation id and a guarantee that any caller-supplied IP is syntactically a valid IPv4/IPv6 address. It is pure policy — no JavaScript — using two RaiseFault policies gated on regex conditions and one AssignMessage.

Where it sits

This is an early, cheap, deterministic gate. It runs after transport-level preflight and before any cryptographic work, so that every later policy — and every error response — can attach the same x-fapi-interaction-id for end-to-end tracing.

The bundle declares three policies and a single default sub-flow:

NameTypePurpose
RF-InvalidInteractionId RaiseFault Return 400 when x-fapi-interaction-id is present but is not a UUID (8-4-4-4-12 hex form). Echoes the offending value back in the fault for traceability.
AM-InteractionId AssignMessage Echo the caller's interaction id when present, otherwise fall back to the built-in {messageid}. Publishes the resolved value on fapi.interaction.id and re-sets it on the request header.
RF-InvalidCustomerIp RaiseFault Return 400 when x-fapi-customer-ip-address is present but is not a valid IPv4 or IPv6 literal. Uses the now-normalized fapi.interaction.id in the fault.

Step order and conditions

The default sub-flow runs the three steps top to bottom. Both RaiseFault steps are conditional — they only fire when the relevant header is both present and malformed; AM-InteractionId is unconditional.

Reject a malformed interaction id (conditional)

RF-InvalidInteractionId fires only when the header is non-null and fails the UUID regex. Crucially the condition uses != null, so an absent header is allowed through — FAPI permits the caller to omit it and have the gateway mint one. The regex is matched with the Apigee JavaRegex operator !~~ (does-not-match).

Echo or generate, then publish (unconditional)

AM-InteractionId always runs. It resolves fapi.interaction.id from the request header first; because IgnoreUnresolvedVariables is true, an absent header makes the policy fall through to the <Value>{messageid}</Value> template. The resolved id is then written back onto the request header so upstreams and later policies see a single, consistent value.

Reject a malformed customer IP (conditional)

RF-InvalidCustomerIp fires only when x-fapi-customer-ip-address is non-null and matches neither the IPv4 nor the IPv6 pattern. By the time this step can fault, fapi.interaction.id is already populated, so its error response carries the normalized correlation id.

sharedflows/default.xmlXML
<!-- 1. Reject a malformed x-fapi-interaction-id only when the header is present. -->
<Step>
  <Name>RF-InvalidInteractionId</Name>
  <Condition>(request.header.x-fapi-interaction-id != null) and
    (request.header.x-fapi-interaction-id !~~
     "^[0-9a-fA-F]{8}-[0-9a-fA-F]{4}-[0-9a-fA-F]{4}-[0-9a-fA-F]{4}-[0-9a-fA-F]{12}$")</Condition>
</Step>
<!-- 2. Echo or generate the id, publish fapi.interaction.id, set the request header. -->
<Step>
  <Name>AM-InteractionId</Name>
</Step>
<!-- 3. Reject a malformed x-fapi-customer-ip-address only when present (IPv4 or IPv6). -->
<Step>
  <Name>RF-InvalidCustomerIp</Name>
  <Condition>(request.header.x-fapi-customer-ip-address != null) and
    (request.header.x-fapi-customer-ip-address !~~ "...IPv4|IPv6 regex...")</Condition>
</Step>

The echo-or-generate trick

The interaction-id normalization is the heart of this flow. A single AssignVariable with both a Ref and a Value gives you a try-header-then-fall-back-to-generated semantic for free:

policies/AM-InteractionId.xmlXML
<AssignMessage name="AM-InteractionId">
  <IgnoreUnresolvedVariables>true</IgnoreUnresolvedVariables>
  <AssignVariable>
    <Name>fapi.interaction.id</Name>
    <Ref>request.header.x-fapi-interaction-id</Ref>
    <Value>{messageid}</Value>
  </AssignVariable>
  <Set><Headers>
    <Header name="x-fapi-interaction-id">{fapi.interaction.id}</Header>
  </Headers></Set>
  <AssignTo createNew="false" type="request"/>
</AssignMessage>
Why {messageid} and not a fresh UUID policy?

Apigee's built-in {messageid} is a guaranteed-unique id minted per request in UUID form, so it satisfies the FAPI requirement to generate an interaction id when the caller omits one — without adding a GenerateUUID step. If your OBIE conformance suite insists on a strictly RFC 4122-versioned UUID, swap the <Value> for a generated UUID variable (see knobs below).

FAPI relevance

FAPI 1.0 Advanced and the UK Open Banking Read/Write standard require ASPSPs to support these correlation and forwarding headers. This flow encodes those rules at the edge:

  • Play-back-or-generate. The profile says: if the request carries an x-fapi-interaction-id, the ASPSP must play it back on the response; otherwise it must set one. Publishing to fapi.interaction.id lets sf-error-handling and sf-logging echo the same value on every response and log line.
  • Optional, but validated when present. Both headers are optional on the wire, so the gate never rejects a missing header — it only rejects a syntactically broken one. This matches the FAPI stance that x-fapi-customer-ip-address is supplied by the client only when a customer is present.
  • OBIE-shaped errors. Both faults return the UK OBIE error envelope with ErrorCode UK.OBIE.Header.Invalid, a Code of 400, and the interaction id in the Id field.
policies/RF-InvalidInteractionId.xml — FaultResponse payloadJSON
{
  "Code": "400",
  "Id": "{request.header.x-fapi-interaction-id}",
  "Message": "Request failed FAPI header validation.",
  "Errors": [
    {
      "ErrorCode": "UK.OBIE.Header.Invalid",
      "Message": "The x-fapi-interaction-id header must be a valid UUID (RFC 4122 8-4-4-4-12 hexadecimal form)."
    }
  ]
}
Two different id sources in the two faults

RF-InvalidInteractionId runs before AM-InteractionId, so it can only echo the raw, invalid {request.header.x-fapi-interaction-id} back in Id. RF-InvalidCustomerIp runs after normalization, so it uses the clean {fapi.interaction.id}. This is intentional, not an inconsistency.

Configuration knobs for production

This flow ships with sensible defaults and very few placeholders. The items below are the decisions you should consciously confirm before promoting beyond the eval environment.

KnobWhereDefault & production guidance
UUID strictness default.xml regex Default accepts any 8-4-4-4-12 hex string. For strict RFC 4122 also constrain the version nibble ([1-5]) and variant ([89abAB]) if your conformance suite requires it.
Generated id format AM-InteractionId <Value> Default {messageid}. To force a freshly generated v4 UUID, add a GenerateUUID (or KVM/JS) producer and reference its output variable here instead.
Customer-IP family RF-InvalidCustomerIp regex Accepts dotted-quad IPv4 and the full IPv6 grammar. Tighten or drop IPv6 only if your estate is IPv4-only; the regex does not currently accept IPv6 zone-id (%eth0) suffixes.
Error namespace both RaiseFault payloads UK.OBIE.Header.Invalid — keep aligned with your OBIE error catalogue and the envelope used by sf-error-handling.

Contract: what downstream gets

Always true after this flow

fapi.interaction.id is set to a valid id (echoed or minted), and request.header.x-fapi-interaction-id equals it. Later flows and the response can rely on a single correlation id.

Never happens

The flow never rejects an absent header and never adds a customer-IP value of its own — x-fapi-customer-ip-address is left exactly as the caller sent it (or omitted).
