Build decisions

Apigee Docs was co-designed against a single source of truth — BLUEPRINT.md — which locks six decisions and records the exact trial-org constraints that shaped them. The dominant constraint is IAM: the build account holds only roles/apigee.admin, so the chassis is fully deployable by hand with a personal token, but the GitHub Actions deploy job is deliberately inert behind a variable guard until a one-time IAM elevation creates the service account and Workload Identity pool. Everything else — a 10-flow chassis plus one reference proxy, an AssignMessage mock with no backend, conditional security steps that keep the demo GET flowing — follows from running on a shared, time-boxed eval org with the fewest possible permissions. This page is the honest log: what was locked, why, and every gotcha solved on the real deploy.

Read this page if you want the reasoning behind the artifacts. The flow pages tell you what each control does; this one tells you why the project is shaped the way it is, and what was actually fought during the build.

The co-design approach

Before any policy XML was written, the project and its owner agreed a blueprint and froze it in the private repo as BLUEPRINT.md with a single rule: this file is the source of truth for why the project is shaped the way it is. Decisions were taken once, written down with their rationale, and then implemented to the letter — so the artifacts and the documentation never drift from the intent. Every claim on this site is traceable back to a locked decision or to a verified environment fact.

Two repos, on purpose

The split is itself a locked decision: a private code-and-pipeline repo (apigee-bank-chassis) and a public docs repo (apigee-docs). Deployable artifacts and CI/CD stay private; the how-it-works is open for learning. This page lives in the public repo and quotes the private one verbatim.

Engineering log — gotchas solved

The honest part. Each item below is a real obstacle hit during the build or the deploy, and the decision that resolved it.

Shared flows must exist before the proxy

The reference proxy's PreFlow is a list of FlowCallout steps that reference shared flows by name. If those flows are not already present in the target environment, the proxy import fails. deploy.sh therefore deploys all ten flows first, in order, and only then the proxy.

tools/deploy.shbash
# Shared flows are deployed FIRST (the reference proxy's FlowCallouts
# require them to already exist in the target environment), then the proxy.
for sf in "${SHARED_FLOWS[@]}"; do
  apigeecli sharedflows create bundle -n "$sf" ... --ovr --wait
done
apigeecli apis create bundle -n "bank-accounts-v1" ... --ovr --wait

The crypto library has to ride inside the bundle

Two flows do JOSE work in JavaScript — sf-jwe (JWE decrypt) and sf-mtls-binding (certificate-thumbprint check). Apigee JS callouts cannot fetch dependencies at runtime, so the 342 KB jsrsasign-all-min.js must be a bundled resource. It is vendored once in tools/vendor/ and staged into each bundle's resources/jsc/ at deploy time, then loaded ahead of the policy's own script via an ordered IncludeURL.

sf-jwe · policies/JS-DecryptJWE.xmlXML
<Javascript name="JS-DecryptJWE" timeLimit="5000">
  <IncludeURL>jsc://jsrsasign-all-min.js</IncludeURL>  <!-- loads first -->
  <ResourceURL>jsc://jwe.js</ResourceURL>
</Javascript>
Include order is load-bearing

IncludeURL resources are evaluated before the ResourceURL script, so jsrsasign is defined before jwe.js / certBinding.js run. Reversing them produces a runtime ReferenceError, not an import error — it only fails when the conditional step actually fires.

You cannot write -- inside an XML comment

The VJWT-AccessToken policy documents how to swap the JWKS URI for an inline PEM key for offline testing. A PEM armor line — the BEGIN/END PUBLIC KEY banner — contains a run of dashes, and a double hyphen is illegal inside an XML comment. The comment therefore omits the dash banner and says so explicitly. This is exactly the class of error that validate.sh catches before Apigee ever sees the bundle.

A policy's root name= must equal its filename

Apigee binds a policy file to the bundle by its root-element name attribute, and that must match the file's basename or the import is rejected with an unhelpful error. validate.sh enforces this statically — alongside step ↔ policy ↔ manifest cross-references — so the failure surfaces in CI, not on a precious trial-org deploy.

tools/validate.sh — cross-reference checkbash
# each policy file's root name attr must equal its filename
fn=$(basename "$pf" .xml)
grep -qE "name=\"$fn\"" "$pf" || echo "  ✗ root name != filename"

Conditional security so the demo GET still flows

Three of the strongest controls are conditioned on the request, not always-on: FC-JwsValidation runs only when x-jws-signature is present; FC-MtlsBinding only with a cnf claim or forwarded client cert; FC-Jwe only for JOSE/JWE content types. This lets the no-body demo GET pass while leaving every control wired and ready. The trade-off is documented in the policy itself — production write/payment endpoints MUST enforce presence.

This is a demo affordance, not a production posture

A request with no signature today passes the JWS step because the step is conditional, not because the signature was checked. Before production, the conditions on write and payment endpoints must be inverted to reject when the proof is absent.

Zero external dependencies for the target

Because the build account has no permission to stand up backends or target servers, the reference proxy ships a RouteRule name="no-target" and answers from an AssignMessage that returns a mock OB Account list. The whole chassis is therefore demonstrable end-to-end with nothing but roles/apigee.admin.

bank-accounts-v1 · proxies/default.xmlXML
<RouteRule name="no-target"/>          <!-- no backend; mock answers -->
<DefaultFaultRule name="default-fault">
  <Step><Name>FC-ErrorHandling</Name></Step>
</DefaultFaultRule>

The IAM blocker — and why the deploy job is inert

The single largest constraint. The signed-in account holds only roles/apigee.admin — enough to import and deploy with a personal access token, but not enough to create the service account and Workload Identity pool that keyless GitHub Actions auth requires. So the pipeline is split in two: ci-validate.yml runs on every push with no credentials, while deploy-apigee.yml is gated off until a human performs the one-time elevation.

.github/workflows/deploy-apigee.ymlYAML
permissions:
  id-token: write   # keyless WIF — no SA JSON keys ever stored
jobs:
  deploy:
    # INERT until IAM elevation + WIF setup is complete
    if: ${{ vars.APIGEE_DEPLOY_ENABLED == 'true' }}

Activation is a documented runbook, not guesswork: elevate IAM in the console, run tools/setup-wif.sh (it creates the deploy SA, the pool/provider pinned by attribute.repository to the exact repo, and the impersonation binding), then set the four repo variables it prints — including APIGEE_DEPLOY_ENABLED=true. No service-account JSON key is ever created; auth is short-lived OIDC.
