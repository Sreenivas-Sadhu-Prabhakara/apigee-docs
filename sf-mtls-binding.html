Shared flow · RFC 8705 / FAPI

sf-mtls-binding

sf-mtls-binding enforces sender-constrained (certificate-bound) access tokens per RFC 8705. A bound token carries a cnf (confirmation) claim of the form {"x5t#S256":"<base64url>"}. This flow recomputes that thumbprint from the client's mTLS certificate — base64url(SHA-256(DER-encoded leaf cert)) using jsrsasign — and compares it to the claim. Two policies run: JS-VerifyCertBinding does the compute-and-compare and writes cert.binding.match; RF-CertBindingMismatch raises 401 whenever the match is not positively true. The JavaScript fails closed: any missing input, malformed cert, or exception leaves match = false and the request is rejected.

This is the proof-of-possession gate. sf-jwt-validation proves the token is genuine; sf-mtls-binding proves the caller holds the private key the token was issued to. Together they defeat token replay/export: a stolen bearer token is worthless without the matching client certificate and its private key.

Policies in this bundle

Two policies, declared in sf-mtls-binding.xml and sequenced by sharedflows/default.xml. Both depend on the bundled jsrsasign library shipped as a JavaScript resource.

Step order & conditions

This is exactly what sharedflows/default.xml executes. Both steps are conditional — neither runs unconditionally.

JS-VerifyCertBinding — condition

Runs only when (jwt.VJWT-AccessToken.claim.cnf != null) or (request.header.x-client-cert != null). In words: run the check if the verified access token carries a cnf claim, or if a client certificate header is present. If neither is true — an unbound token on a plain connection — the whole flow is a no-op and nothing is rejected.

RF-CertBindingMismatch — condition

Runs only when (cert.binding.match = false). Because the JavaScript initialises cert.binding.match to false and only sets it true on a proven match, every failure path — no cert, no cnf, malformed DER, thumbprint mismatch, or thrown exception — lands here as a 401.

Order matters: this flow runs after sf-jwt-validation

The step condition reads jwt.VJWT-AccessToken.claim.cnf — a variable only populated once the VJWT-AccessToken VerifyJWT policy (in sf-jwt-validation) has run. So although mTLS is terminated at the network edge, the binding verification shared flow must be invoked after token validation, otherwise the cnf branch of the condition can never be true. The fallback x-client-cert != null branch means a cert presented without a bound token still triggers the check — and fails closed with "Access token has no cnf claim", returning 401.

Inside JS-VerifyCertBinding

certBinding.js is a single IIFE that fails closed. Its first act is to set the defaults so any early return or thrown error leaves the request rejectable.

resources/jsc/certBinding.js — fail-closed defaultsJS
// Default to false: fail closed unless we positively prove a match.
context.setVariable('cert.binding.match', false);
context.setVariable('cert.binding.computed', '');
context.setVariable('cert.binding.expected', '');

It then proceeds through four stages. Any failure in a stage records a human-readable reason in cert.binding.error and returns early — leaving match = false.

1 · Locate the client certificate

Tries three sources in priority order, most specific first: request.header.x-forwarded-client-cert, then the native tls.client.raw.cert, then request.header.x-client-cert. On Apigee X behind a Google Cloud external HTTPS load balancer the client cert is not a native TLS variable — the LB forwards it as a header — so the header sources come first, with the native variable kept for hybrid topologies. No cert at all → error "No client certificate found in request."

2 · Read the expected thumbprint

Reads jwt.VJWT-AccessToken.claim.cnf. If absent → "Access token has no cnf claim." The value may already be an object or a JSON string; a string is JSON.parsed (parse failure → "cnf claim is not valid JSON."). It pulls cnf['x5t#S256'] (missing → "cnf claim does not contain x5t#S256."), strips whitespace, and stores it in cert.binding.expected.

3 · Normalise the certificate to DER (hex)

The forwarded cert can arrive in several shapes, handled defensively (see below) and converted to DER hex via jsrsasign — pemtohex() for PEM, b64utohex() for bare base64. Failure to derive bytes → "Unable to derive DER bytes from client certificate."

4 · Hash, compare, decide

Computes SHA-256 over the DER hex with KJUR.crypto.Util.hashHex, encodes it base64url (no padding) with hextob64u, stores it in cert.binding.computed, and sets cert.binding.match = (computed === expected). On a non-match it records "Thumbprint mismatch: computed != cnf x5t#S256."

Certificate-shape normalisation

Stage 3 copes with four real-world encodings of the forwarded leaf certificate before it ever reaches jsrsasign:

  • (a) XFCC / Envoy style. Headers like Cert="<url-encoded PEM>";Chain="…";… — the leaf is extracted with the regex /Cert="?([^";]+)"?/i.
  • (b) URL-encoded PEM. If the string contains % it is decodeURIComponentd (after turning + back into spaces) to restore %0A newlines and %20 spaces.
  • (c) PEM with literal newlines. If it contains -----BEGIN it is converted directly with pemtohex(certStr, 'CERTIFICATE').
  • (d) Bare base64 of the DER. No PEM armor: whitespace is stripped and base64url is normalised to base64 (-→+, _→/) before b64utohex().

The crypto, as written

The load-bearing comparison is four lines. The thumbprint definition is RFC 8705 verbatim: x5t#S256 == base64url( SHA-256( DER-encoded client certificate ) ).

resources/jsc/certBinding.js — compute & compareJS
var digestHex = KJUR.crypto.Util.hashHex(certHex, 'sha256');
var computed  = hextob64u(digestHex); // base64url, no padding
context.setVariable('cert.binding.computed', computed);

var match = (computed === expected);
context.setVariable('cert.binding.match', match);
if (!match) {
  context.setVariable('cert.binding.error',
    'Thumbprint mismatch: computed != cnf x5t#S256.');
}

The whole IIFE is wrapped in try/catch so even an unexpected jsrsasign error fails closed:

resources/jsc/certBinding.js — catch-allJS
catch (e) {
  // Fail closed on any unexpected error.
  context.setVariable('cert.binding.match', false);
  context.setVariable('cert.binding.error',
    'certBinding.js exception: ' + (e && e.message ? e.message : e));
}

Output context variables

cert.binding.match cert.binding.computed cert.binding.expected cert.binding.error

match is the only variable the flow acts on ("true" / "false"). computed, expected and error exist for diagnostics and logging — useful when sf-logging records why a binding check failed.

RF-CertBindingMismatch — the rejection

An OBIE-shaped 401. The interaction id is read from messaging.fapi.interaction.id (set upstream by sf-fapi-headers), not from the raw request header. IgnoreUnresolvedVariables is true so a missing interaction id never turns the fault itself into an error.

FAPI relevance

Certificate-bound access tokens are a core FAPI 1.0 Advanced requirement. They turn a bearer token into a holder-of-key token: possession alone is insufficient — the presenter must also complete the mTLS handshake with the key the token was issued against.

RFC 8705 PoP cnf / x5t#S256 SHA-256 leaf thumbprint Fail-closed OBIE error contract
  • Sender-constrained tokens. A token exfiltrated from logs, a proxy, or a compromised client cannot be replayed against this API without the matching client certificate and its private key.
  • Leaf, not chain. The thumbprint is over the leaf certificate's DER, matching how the authorization server computed x5t#S256 at token issuance — the XFCC Cert= extraction deliberately isolates the leaf from any forwarded chain.
  • Defence in depth. Pairs with sf-jwt-validation (token authenticity) so authenticity and possession are checked by separate, composable flows.
  • Traceability. The 401 echoes x-fapi-interaction-id, keeping rejected calls correlatable end to end.
Why jsrsasign instead of native variables

Apigee X does not expose a built-in cert-thumbprint variable that survives an external HTTPS load balancer, which strips the TLS layer and re-presents the client cert as a forwarded header. Computing SHA-256(DER) in JavaScript with jsrsasign is portable across that topology and across hybrid — the same code works whether the cert arrives as XFCC, URL-encoded PEM, raw PEM, or bare base64.

Configuration knobs

The placeholder and environment-specific values to review before production. Most are header/variable names that must match your deployment topology and your sf-jwt-validation policy names.

Client-cert source headers

The fallback chain in certBinding.js — x-forwarded-client-cert, tls.client.raw.cert, x-client-cert. Set these to the exact header your load balancer forwards the client certificate in; an LB that uses a different header name will look like "No client certificate found".

cnf claim variable

jwt.VJWT-AccessToken.claim.cnf is hard-wired to the VerifyJWT policy named VJWT-AccessToken in sf-jwt-validation. If you rename that policy, update both the JS read and the step <Condition> in default.xml.

Interaction-id variable

The fault reads messaging.fapi.interaction.id. This must match where sf-fapi-headers stores the FAPI interaction id; otherwise the 401 ships an empty x-fapi-interaction-id.

JS time limit

timeLimit="2000" (ms) on JS-VerifyCertBinding. The base64 decode + SHA-256 is fast, but large forwarded chains plus jsrsasign init can take time; tune if you see JavaScript timeout faults under load.

Error contract

The UK.OBIE.Unauthorized code and the two messages in RF-CertBindingMismatch. Keep them aligned with your tenant's published OBIE error catalogue.

jsrsasign resource

jsc://jsrsasign-all-min.js is the vendored crypto library providing KJUR, pemtohex, b64utohex and hextob64u. Pin and review its version as part of dependency hygiene.

Verify the LB actually forwards the leaf cert

The binding check is only as strong as the certificate the proxy receives. If the external HTTPS load balancer is not configured to require client mTLS and forward the leaf certificate, the header will be absent, the JS will report "No client certificate found", and every bound token will 401 — or, worse, if mTLS is optional and unauthenticated traffic skips the check entirely, possession is never proven. Enforce client mTLS at the LB and confirm the forwarded-cert header name before relying on this flow.

Where this fits

mTLS is terminated at the network edge, but the binding verification shared flow is invoked in the request path after sf-jwt-validation — it consumes the parsed cnf claim that VerifyJWT publishes. The cumulative order is:

sf-preflight sf-fapi-headers sf-jwt-validation sf-mtls-binding sf-jws-validation sf-jwe

By the time sf-mtls-binding completes, the request is known to carry a genuine, in-scope access token and to be presented by the holder of the key that token was bound to — the proof-of-possession precondition for everything downstream.
