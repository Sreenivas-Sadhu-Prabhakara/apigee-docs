Shared flow

sf-logging

sf-logging produces a structured, redacted log line for every request without needing a Google Cloud service account. It is a single AssignMessage policy — AM-LogContext — that writes a compact JSON document into the audit.log flow variable (plus a discrete audit.log.interactionId variable), so the record is visible in Apigee trace/debug and analytics under nothing more than roles/apigee.admin. Real shipping to Google Cloud Logging is an optional upgrade: the ML-CloudLogging MessageLogging policy lives outside the deployed bundle because it requires a deploy-time service account holding roles/logging.logWriter. Both paths deliberately omit secrets — the Authorization bearer token and the x-jws-signature detached signature are never logged.

Where it sits

This is the observability tail of the chassis. It is designed to run on the response path, after every other flow has executed, so the log line can capture the resolved correlation id and the final response.status.code for the request — success or fault alike.

The deployed bundle is deliberately minimal: it declares a single policy and one default sub-flow with a single step.

NameTypePurpose
AM-LogContext AssignMessage Build a redacted JSON log document and assign it to the audit.log flow variable; also publish audit.log.interactionId from fapi.interaction.id. SA-free — surfaces in trace and built-in analytics.
ML-CloudLogging MessageLogging not deployed Optional upgrade. Ships the same structured, redacted payload to Google Cloud Logging under projects/…/logs/apigee. Held in optional-cloudlogging/ — see below.
No StatisticsCollector here

The bundle <Description> records a build decision: custom analytics via StatisticsCollector is intentionally omitted because it is not supported by this Apigee org's bundle validator. The discrete audit.log.interactionId variable is the SA-free substitute that can later be promoted into a CustomReport.

What AM-LogContext does, step by step

The default sub-flow has exactly one unconditional step, AM-LogContext. Inside that one policy, two AssignVariable operations run in order.

Assemble the structured log line

An AssignVariable with a <Template> renders a single-line JSON object into audit.log. The fields are message metadata only — interaction id, verb, path, status, API product, proxy and environment — drawn from Apigee context variables. No request body, no headers that could carry a secret.

Publish the correlation id discretely

A second AssignVariable copies fapi.interaction.id (normalized upstream by sf-fapi-headers) into audit.log.interactionId. Keeping the id as its own flat variable — not just embedded in the JSON string — makes it filterable in analytics and trace.

Two policy attributes make this safe to attach unconditionally on the response path: continueOnError="true" means a logging hiccup can never fault the API response, and IgnoreUnresolvedVariables="true" means any context variable that is empty (for example on an early-rejected request) simply renders blank instead of breaking the template.

policies/AM-LogContext.xmlXML
<AssignMessage name="AM-LogContext" continueOnError="true" enabled="true">
  <DisplayName>AM-LogContext</DisplayName>
  <IgnoreUnresolvedVariables>true</IgnoreUnresolvedVariables>
  <!-- 1. Structured, redacted JSON into audit.log -->
  <AssignVariable>
    <Name>audit.log</Name>
    <Template>{"interactionId":"{fapi.interaction.id}","verb":"{request.verb}",
      "path":"{request.uri}","status":"{response.status.code}",
      "apiproduct":"{apiproduct.name}","proxy":"{apiproxy.name}",
      "environment":"{environment.name}"}</Template>
  </AssignVariable>
  <!-- 2. Correlation id as its own filterable variable -->
  <AssignVariable>
    <Name>audit.log.interactionId</Name>
    <Ref>fapi.interaction.id</Ref>
  </AssignVariable>
</AssignMessage>

The <Template> is a single physical line in the source; it is wrapped above only for readability.

The rendered record is a flat, machine-parseable line — every field is a context variable, nothing more:

{

Secret redaction

The most important property of this flow is what it does not contain. Both the SA-free policy and the optional Cloud Logging policy log only message metadata; they never serialize the headers that carry credentials or message-integrity proofs.

Never logged, by design

The payload deliberately omits the Authorization header (the bearer access token) and the x-fapi / x-jws-signature header (the detached JWS that provides message integrity). Logging a bearer token would leak a live credential; logging a signature would weaken the non-repudiation guarantee. If you add fields, keep this rule — log identifiers, never secrets or signatures.

FAPI relevance

FAPI 1.0 Advanced and UK Open Banking demand end-to-end traceability without compromising the credentials and signatures the profile relies on. This flow encodes both halves of that requirement:

  • One correlation id, everywhere. Every log record carries fapi.interaction.id, the same value that sf-fapi-headers echoes on the response and sf-error-handling stamps on faults. A single id threads the client request, the gateway logs, and the response header.
  • Status captured on the response path. Because the policy reads response.status.code, it is intended to run after the response is set, so the log line reflects the real outcome — including faulted requests, thanks to continueOnError.
  • Credential and signature hygiene. Redacting the bearer token and the detached JWS keeps logs from becoming a secondary attack surface — a recurring FAPI conformance and security-review concern.

Optional upgrade: ship to Google Cloud Logging

The SA-free path gives you trace-visible structured logs today. To persist and query them centrally, enable ML-CloudLogging — a real Cloud Logging sink that was lifted out of the bundle for one concrete reason.

Why it is not in the deployed bundle

Apigee X requires a service-account identity whenever a deployed revision contains a Cloud-integrated policy (CloudLogging, GoogleAuthentication, …). Deploying a bundle that contains ML-CloudLogging without one fails with:

Deployment of sf-logging requires a service account identity, but one was not provided.

With only roles/apigee.admin you cannot yet create or actAs a service account, so the policy is parked in optional-cloudlogging/ until the IAM elevation in Phase 5.

The optional policy mirrors the SA-free payload exactly, then adds a Cloud Logging target and two indexed labels:

optional-cloudlogging/ML-CloudLogging.xmlXML
<MessageLogging name="ML-CloudLogging">
  <CloudLogging>
    <LogName>projects/your-apigee-project/logs/apigee</LogName>
    <Message contentType="application/json">{"interactionId":"{fapi.interaction.id}",
      "verb":"{request.verb}","path":"{request.uri}","status":"{response.status.code}",
      "apiproduct":"{apiproduct.name}","proxy":"{apiproxy.name}",
      "environment":"{environment.name}"}</Message>
    <ResourceType>api</ResourceType>
    <Labels>
      <Label><Key>interactionId</Key><Value>{fapi.interaction.id}</Value></Label>
      <Label><Key>proxy</Key><Value>{apiproxy.name}</Value></Label>
    </Labels>
  </CloudLogging>
  <IgnoreUnresolvedVariables>true</IgnoreUnresolvedVariables>
</MessageLogging>

Turning it on is a three-move sequence, documented in optional-cloudlogging/README.md and gated on the IAM work in Phase 5:

Create a runtime SA and grant log-writer

Create apigee-runtime and bind roles/logging.logWriter so the deployed revision's identity may write Cloud Logging entries.

Move the policy back into the bundle

mv optional-cloudlogging/ML-CloudLogging.xml sharedflowbundle/policies/, add <Policy>ML-CloudLogging</Policy> to sf-logging.xml, and add a <Step><Name>ML-CloudLogging</Name></Step> to sharedflows/default.xml.

Deploy with the SA identity

Re-deploy passing the service account with apigeecli's -s flag, so Apigee attaches the identity the Cloud-integrated policy demands.

Enable Cloud Logging (after IAM elevation)SHELL
$ gcloud iam service-accounts create apigee-runtime \
    --project your-apigee-project

$ gcloud projects add-iam-policy-binding your-apigee-project \
    --member "serviceAccount:apigee-runtime@your-apigee-project.iam.gserviceaccount.com" \
    --role roles/logging.logWriter

$ apigeecli sharedflows deploy -n sf-logging -e eval --ovr --wait \
    -s apigee-runtime@your-apigee-project.iam.gserviceaccount.com \
    -o your-apigee-project -t "$(gcloud auth print-access-token)"

Configuration knobs for production

The flow ships ready to deploy with no required edits in the SA-free path. The items below are the decisions to confirm before promoting beyond eval or enabling the Cloud Logging sink.

KnobWhereDefault & production guidance
GCP project / log name ML-CloudLogging <LogName> Hard-coded to projects/your-apigee-project/logs/apigee. Repoint to your project and log id before enabling the sink.
Runtime service account deploy -s flag apigee-runtime@… with roles/logging.logWriter. Required for any Cloud-integrated policy; not needed for the SA-free default.
Log fields AM-LogContext <Template> / ML-CloudLogging <Message> Metadata only by default. Add fields if needed, but never the Authorization bearer token or the x-jws-signature value — keep the two payloads in sync.
Indexed labels ML-CloudLogging <Labels> interactionId and proxy. Add labels for fields you intend to filter on at scale in Cloud Logging.
Custom analytics (not present) StatisticsCollector is intentionally omitted (org validator limitation). Re-introduce only if your org's bundle validator accepts it.

Contract: what downstream gets

Always true after this flow

audit.log holds a single-line JSON record and audit.log.interactionId holds the correlation id — both visible in trace and analytics, under roles/apigee.admin alone.

Never happens

No credential or signature is written to any log, the API response is never faulted by a logging error (continueOnError="true"), and no service account is required unless you opt into Cloud Logging.

Source of truth: shared-flows/sf-logging/sharedflowbundle/ — one policy (AM-LogContext) wired by sharedflows/default.xml; the optional sink and enablement steps live in shared-flows/sf-logging/optional-cloudlogging/ (ML-CloudLogging.xml + README.md).
