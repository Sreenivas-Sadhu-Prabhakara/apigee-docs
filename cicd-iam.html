CI/CD & IAM

The chassis ships two GitHub Actions workflows. ci-validate.yml is active today: it runs tools/validate.sh (xmllint well-formedness + cross-reference integrity) on every push and PR, needs no credentials, and is green on every commit. deploy-apigee.yml is inert by design: it deploys to Apigee X over keyless Workload Identity Federation, but an if: vars.APIGEE_DEPLOY_ENABLED == 'true' guard skips it until activated. It is gated because the signed-in account holds only roles/apigee.admin — enough to deploy by hand, but not enough to create the deploy service account or the WIF pool the pipeline needs. Activation is a one-time, self-service IAM elevation followed by tools/setup-wif.sh. No service-account JSON keys are ever created or stored.

CI/CD here is split along a single fault line: validation is free and unconditional, deployment requires Google Cloud identity. The first half runs from day one; the second half stays dormant behind a feature flag until the operator grants themselves the IAM they don't yet have.

The two workflows

Both live under .github/workflows/. They are mutually exclusive in concern — one validates artifacts, one ships them — and together they cover the full path from commit to a deployed Apigee environment.

WorkflowTriggerPurpose
ci-validate.yml every push, every pull_request, plus workflow_dispatch active now Runs tools/validate.sh — the same well-formedness + cross-reference checks Apigee enforces on import. No GCP credentials required.
deploy-apigee.yml workflow_dispatch + push of a v* tag inert Deploys shared flows then the proxy via tools/deploy.sh over WIF. Skipped until APIGEE_DEPLOY_ENABLED == 'true'.

ci-validate.yml — active today

A single validate job on ubuntu-latest: check out, install libxml2-utils (for xmllint), then run tools/validate.sh. There is no Google Cloud step, no secret, and nothing to gate — which is exactly why it runs on every push without ceremony.

.github/workflows/ci-validate.ymlYAML
on:
  push:
  pull_request:
  workflow_dispatch:

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install xmllint
        run: sudo apt-get update && sudo apt-get install -y libxml2-utils
      - name: Validate Apigee bundles (xmllint + cross-references)
        run: bash tools/validate.sh

tools/validate.sh exits non-zero on the first defect, so a broken bundle turns the commit red. It enforces three invariants across every shared flow and the reference proxy:

  • XML well-formedness — xmllint --noout on every *.xml under shared-flows/ and apiproxies/.
  • Step ↔ policy file ↔ manifest integrity — every <Name> referenced in a flow has a matching policies/<name>.xml and a <Policy> entry in the bundle manifest, and vice-versa.
  • Filename equals root name attribute — each policy file's root element name="…" matches its filename, the constraint Apigee imposes on import.
Validation mirrors Apigee's import checks

The point of ci-validate.yml is to fail fast and free. It reproduces the structural checks Apigee runs at import time, so a malformed bundle is caught at PR review — long before anyone spends a deploy (and trial-clock minutes) discovering it the hard way.

deploy-apigee.yml — keyless deploy via WIF

The deploy workflow is deliberately conservative for a shared eval org on a trial clock: it never fires on a plain push to main. It runs only on manual workflow_dispatch (with an environment input defaulting to eval) or when a v* tag is pushed.

.github/workflows/deploy-apigee.yml — trigger, permissions, guardYAML
on:
  workflow_dispatch:
    inputs:
      environment:
        default: eval
        required: true
  push:
    tags:
      - 'v*'

permissions:
  contents: read
  id-token: write   # required for Workload Identity Federation (keyless auth)

jobs:
  deploy:
    runs-on: ubuntu-latest
    if: ${{ vars.APIGEE_DEPLOY_ENABLED == 'true' }}

The id-token: write permission is the load-bearing line: it lets the job mint a short-lived GitHub OIDC token, which is the entire basis of the keyless model below. The if: guard is what keeps the job dormant — with the repo variable unset, every run resolves to "skipped" rather than "failed".

How the keyless model works

No JSON key files exist anywhere — not in the repo, not in GitHub Secrets. Instead the job presents its GitHub-issued OIDC token to Google's Security Token Service, which (because the WIF provider trusts this specific repo) exchanges it for a short-lived access token scoped to the apigee-deployer service account.

GitHub mints an OIDC token

With id-token: write set, the runner obtains a signed JWT from token.actions.githubusercontent.com asserting the workflow's identity — including its repository claim.

WIF exchanges it for Google credentials

google-github-actions/auth@v2 sends that token to STS against vars.WIF_PROVIDER. The provider's attribute condition assertion.repository=='…/apigee-bank-chassis' must hold, or the exchange is refused.

It impersonates the deploy SA

The exchanged identity is bound to apigee-deployer via roles/iam.workloadIdentityUser, yielding a short-lived token printed by gcloud auth print-access-token. No long-lived key is ever materialised.

apigeecli deploys flows then proxy

The job installs apigeecli, exports that token as APIGEE_TOKEN, and runs tools/deploy.sh — which deploys all ten shared flows first (the proxy's FlowCallouts require them to pre-exist), then bank-accounts-v1.

.github/workflows/deploy-apigee.yml — stepsYAML
      - name: Authenticate to Google Cloud (Workload Identity Federation)
        uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: ${{ vars.WIF_PROVIDER }}
          service_account: ${{ vars.WIF_SERVICE_ACCOUNT }}

      - name: Set up gcloud
        uses: google-github-actions/setup-gcloud@v2

      - name: Install apigeecli
        run: curl -s https://raw.githubusercontent.com/apigee/apigeecli/main/downloadLatest.sh | bash

      - name: Deploy shared flows then proxy
        run: |
          export PATH="$PATH:$HOME/.apigeecli/bin"
          export APIGEE_TOKEN="$(gcloud auth print-access-token)"
          ./tools/deploy.sh

Why the deploy job is gated

This is the central design fact of the page, so state it plainly: the account that owns this project holds only roles/apigee.admin on your-apigee-project. That role is sufficient to push bundles with your own token — which is exactly how Phase 4 deployed to eval by hand — but it cannot:

  • create the apigee-deployer service account (needs roles/iam.serviceAccountAdmin),
  • create the Workload Identity Pool and OIDC provider (needs roles/iam.workloadIdentityPoolAdmin),
  • grant project-level IAM bindings to that SA (needs roles/resourcemanager.projectIamAdmin),
  • enable the STS / IAM Credentials APIs (needs roles/serviceusage.serviceUsageAdmin).

Because the prerequisites cannot be created from the current permission set, the pipeline cannot authenticate. Rather than ship a workflow that fails on every run, the chassis ships it skipped behind APIGEE_DEPLOY_ENABLED and documents the one-time elevation that flips it on.

Elevation is a deliberate, human-in-the-loop act

The pipeline does not — and structurally cannot — escalate its own privileges. A person must grant the IAM in the Cloud Console first. This keeps the privileged step auditable and out of automation, which matters for a project whose whole point is a defensible security posture.

Activating the pipeline — one time

Four steps, performed once. The first is the manual IAM grant; the rest are scripted or are simple variable registration.

Elevate your IAM (Cloud Console)

Grant your account roles/owner on your-apigee-project, or the minimal bundle: iam.serviceAccountAdmin, iam.workloadIdentityPoolAdmin, resourcemanager.projectIamAdmin, serviceusage.serviceUsageAdmin.

Run tools/setup-wif.sh

Creates the deploy SA, the WIF pool + GitHub OIDC provider bound to this repo, and the impersonation binding. It then prints the exact gh variable set commands to run.

Set the four repo variables

Register WIF_PROVIDER, WIF_SERVICE_ACCOUNT, APIGEE_ORG, and APIGEE_DEPLOY_ENABLED=true — the last of which un-skips the deploy job.

Deploy

Run gh workflow run "Deploy to Apigee X" or push a v* tag.

What tools/setup-wif.sh actually does

The script is set -euo pipefail and idempotent (creation calls tolerate "already exists"). Against your-apigee-project (project number PROJECT_NUMBER) it performs four blocks of work.

ActionResourcePurpose
Enable APIs iamcredentials, sts, apigee Turn on the services the token exchange and deploy depend on.
Create SA + grant roles apigee-deployer Grants roles/apigee.apiAdminV2 and roles/apigee.environmentAdmin — the deploy-scoped permissions, narrower than the human's apigee.admin.
Create WIF pool + provider github-pool / github-provider OIDC provider trusting token.actions.githubusercontent.com, with attribute condition assertion.repository=='…/apigee-bank-chassis' so only this repo can exchange.
Bind repo identity to SA roles/iam.workloadIdentityUser Lets the repo's principalSet://…/attribute.repository/… impersonate apigee-deployer — the link that makes keyless auth resolve.
tools/setup-wif.sh — WIF pool + repo-scoped OIDC providerbash
gcloud iam workload-identity-pools providers create-oidc "$PROVIDER" \
  --location global --workload-identity-pool "$POOL" \
  --display-name "GitHub provider" \
  --attribute-mapping "google.subject=assertion.sub,attribute.repository=assertion.repository" \
  --attribute-condition "assertion.repository=='$REPO'" \
  --issuer-uri "https://token.actions.githubusercontent.com" || true

On success the script prints the variable-registration commands verbatim:

tools/setup-wif.sh — printed outputshell
$ gh variable set WIF_PROVIDER          -b "$WIF_PROVIDER"   -R "$REPO"
$ gh variable set WIF_SERVICE_ACCOUNT   -b "$SA_EMAIL"       -R "$REPO"
$ gh variable set APIGEE_ORG            -b "$PROJECT"        -R "$REPO"
$ gh variable set APIGEE_DEPLOY_ENABLED -b "true"            -R "$REPO"

$ gh workflow run "Deploy to Apigee X" -R "$REPO"

No keys — ever

Authentication is short-lived OIDC end to end. No service-account JSON key is created, downloaded, or stored in GitHub Secrets. If you find yourself generating a key file to make this work, you have left the intended design — re-check the WIF binding instead.
