Deployment & testing

Deploy with apigeecli, authenticated by a short-lived gcloud access token. tools/deploy.sh pushes the ten shared flows FIRST, then the bank-accounts-v1 proxy — because the proxy's FlowCallout policies resolve shared flows by name at import time, so each one must already exist in the target environment or the proxy deploy fails. Gate every push with tools/validate.sh (static XML + cross-reference checks), then prove the chassis end-to-end with two live curl smoke tests against the eval host your-eval-host: an unauthenticated request returns 401 UK.OBIE.Unauthorized, and a malformed x-fapi-interaction-id returns 400 UK.OBIE.Field.Invalid. Two org-specific realities shaped the bundle: StatisticsCollector analytics is omitted (this org's bundle validator rejects it), and native CloudLogging ships out-of-band because it needs a deploy-time service account.

There is no Maven plugin, no apigee-deploy magic, and no long-lived service-account key in this path. Deployment is a single idempotent Bash script driving the official apigeecli binary with a token you mint locally — the same primitives the CI pipeline uses, minus Workload Identity Federation.

Prerequisites

Three things must be in place before deploy.sh will run: the apigeecli binary, an access token, and (optionally) the org/env overrides. The defaults target the eval environment used throughout this documentation.

install + authenticatebash
# 1. Install apigeecli — the official installer drops the binary at
#    ~/.apigeecli/bin/apigeecli, which is exactly the path deploy.sh probes.
curl -L https://raw.githubusercontent.com/apigee/apigeecli/main/downloadLatest.sh | sh

# 2. Authenticate to GCP once; the token is read at deploy time.
gcloud auth login

# 3. Point at the target org/env (defaults shown — export to override).
export APIGEE_ORG=your-apigee-project
export APIGEE_ENV=eval
export APIGEE_TOKEN=$(gcloud auth print-access-token)

Every value above has a fallback baked into the script, so on the default org/env you can run ./tools/deploy.sh with no exports at all — APIGEE_TOKEN falls back to gcloud auth print-access-token at runtime.

A gcloud token, never a downloaded key

The local path uses an ephemeral user token from gcloud auth print-access-token — nothing is persisted to disk. The CI path replaces this with Workload Identity Federation (no key at all). At no point does this repository carry a service-account JSON key. See CI/CD & IAM.

Validate before you deploy

tools/validate.sh runs the same two classes of check Apigee enforces on import, but locally and in milliseconds — so a broken bundle never costs you a round trip to the management API. It exits non-zero on any failure, which is why CI runs it as a gate.

1. XML well-formedness

Every *.xml under shared-flows/ and apiproxies/ is parsed with xmllint --noout. A single malformed policy fails the run and prints the parser error.

2. Cross-reference integrity

For each bundle it proves the three-way contract holds: every <Step> name in default.xml has a matching policy file and a <Policy> entry in the manifest; every manifest policy has a file; and every policy file's root name= attribute equals its filename.

static gatebash
./tools/validate.sh
# >> XML well-formedness (xmllint)
# >> Cross-reference integrity (steps <-> policy files <-> manifest)
# ✓ ALL CHECKS PASSED

Deploy order — shared flows first, then the proxy

tools/deploy.sh is set -euo pipefail and runs four phases. The ordering is not cosmetic: it is the single most important operational fact about this chassis.

Resolve org / env / token

Reads APIGEE_ORG, APIGEE_ENV, APIGEE_TOKEN with the defaults above; TOKEN falls back to gcloud auth print-access-token if unset. CLI resolves to $APIGEECLI or ~/.apigeecli/bin/apigeecli.

Stage the vendored crypto library

Copies tools/vendor/jsrsasign-all-min.js into the resources/jsc directory of sf-jwe and sf-mtls-binding if it is not already present — those two bundles' JavaScript policies depend on it. The check is idempotent, so re-runs are no-ops.

Deploy the ten shared flows, in dependency order

Iterates the SHARED_FLOWS array (sf-preflight → sf-fapi-headers → sf-jwt-validation → sf-jws-validation → sf-jwe → sf-mtls-binding → sf-security-headers → sf-traffic-management → sf-error-handling → sf-logging), creating and deploying each bundle with --ovr --wait.

Deploy the reference proxy

Only now is bank-accounts-v1 imported and deployed. Its FlowCallouts (FC-Preflight, FC-FapiHeaders, …) can now resolve every shared flow they reference.

Why shared flows must go first

The proxy is deliberately thin: its PreFlow is a chain of FlowCallout policies that invoke shared flows by name. Apigee validates those references when the proxy revision is imported — if a referenced shared flow does not yet exist in the environment, the proxy import is rejected. Deploying the chassis first guarantees the names resolve. Deploy the proxy first and you get a hard failure, not a silent gap.

Both phases use the same apigeecli shape. create bundle with -e imports a new revision and deploys it; --ovr overwrites the existing deployment, --wait blocks until the deployment reaches a terminal state (failing the script on error), and --disable-check --no-warnings quiet the CLI's self-update check and non-fatal notices.

tools/deploy.sh — the two deploy callsbash
# Phase 3 — every shared flow, in order:
for sf in "${SHARED_FLOWS[@]}"; do
  "$CLI" sharedflows create bundle \
    -n "$sf" -f "$ROOT/shared-flows/$sf/sharedflowbundle" \
    -e "$ENV" --ovr --wait \
    -o "$ORG" -t "$TOKEN" --disable-check --no-warnings
done

# Phase 4 — the reference proxy, last:
"$CLI" apis create bundle \
  -n "bank-accounts-v1" -f "$ROOT/apiproxies/bank-accounts-v1/apiproxy" \
  -e "$ENV" --ovr --wait \
  -o "$ORG" -t "$TOKEN" --disable-check --no-warnings

Live smoke tests

Two curl calls against the eval host exercise the full PreFlow ordering and prove the chassis is wired end-to-end. The proxy's base path is /open-banking/v1.1/accounts and the accounts resource is /accounts, so the live URL is https://your-eval-host/open-banking/v1.1/accounts/accounts. Both responses are normalized by FC-ErrorHandling in the proxy's DefaultFaultRule, which is why the wire result differs from the raw RaiseFault body.

GET /open-banking/v1.1/accounts/accounts

Test 1 — unauthenticated request → 401

A valid x-fapi-interaction-id but no Authorization: Bearer header. The request clears FC-Preflight and FC-FapiHeaders, then FC-JwtValidation raises RF-MissingToken. The error handler maps the 401 status to the OB contract.

smoke test 1 — missing tokenbash
curl -i https://your-eval-host/open-banking/v1.1/accounts/accounts \
  -H "x-fapi-interaction-id: $(uuidgen)"
expected — HTTP/1.1 401 UnauthorizedJSON
{
  "Code": "401",
  "Id": "<your interaction id>",
  "Message": "The access token is missing, invalid, or expired",
  "Errors": [
    {
      "ErrorCode": "UK.OBIE.Unauthorized",
      "Message": "The access token is missing, invalid, or expired"
    }
  ]
}

Test 2 — malformed FAPI header → 400

An x-fapi-interaction-id that is not an RFC 4122 UUID. Preflight passes (a header-only GET), but FC-FapiHeaders raises RF-InvalidInteractionId before token validation ever runs — header hygiene is enforced ahead of identity.

smoke test 2 — bad interaction idbash
curl -i https://your-eval-host/open-banking/v1.1/accounts/accounts \
  -H "x-fapi-interaction-id: not-a-uuid"
expected — HTTP/1.1 400 Bad RequestJSON
{
  "Code": "400",
  "Id": "<normalized interaction id>",
  "Message": "A request field is missing or invalid",
  "Errors": [
    {
      "ErrorCode": "UK.OBIE.Field.Invalid",
      "Message": "A request field is missing or invalid"
    }
  ]
}
TestRejected byResult
No bearer token, valid headers FC-JwtValidation → RF-MissingToken 401 UK.OBIE.Unauthorized
Malformed x-fapi-interaction-id FC-FapiHeaders → RF-InvalidInteractionId 400 UK.OBIE.Field.Invalid
The wire code is the normalized code, not the RaiseFault code

RF-InvalidInteractionId raises a body carrying UK.OBIE.Header.Invalid, but the DefaultFaultRule re-runs the request through FC-ErrorHandling. errorMap.js classifies the fault by its status code (400) — the RaiseFault fault name does not match the keyword branches — so it remaps to UK.OBIE.Field.Invalid and AM-ErrorResponse rewrites the response. The client sees one consistent OB error shape regardless of which policy first tripped.

Two deployment gotchas

Both are real constraints discovered against this specific Apigee X org, and both are visible in the source. They explain why sf-logging looks smaller than you might expect.

StatisticsCollector is not supported by this org

The original design called for a StatisticsCollector policy in sf-logging to push custom analytics dimensions. This org's bundle validator rejects that policy on import, so it was removed entirely rather than left to fail every deploy. The omission is documented in the bundle's own description.

shared-flows/sf-logging/sharedflowbundle/sf-logging.xmlXML
<Description>Structured observability: Google Cloud Logging with secret
redaction for the FAPI bank API chassis. (Custom analytics via
StatisticsCollector is intentionally omitted — not supported by this
Apigee org's bundle validator.)</Description>
Don't reintroduce it blindly

If you target an org that does support StatisticsCollector, add it back to sf-logging — but re-run tools/validate.sh and a real deploy first. The validator's policy support varies by org provisioning tier, and a rejected policy fails the whole bundle, not just that step.

CloudLogging requires a deploy-time service account

The native MessageLogging + CloudLogging policy (ML-CloudLogging) is not in the deployed bundle. It lives in sf-logging/optional-cloudlogging/. Apigee X demands a runtime service-account identity whenever a deployed revision contains a Cloud-integrated policy; without one the deploy fails with:

Deployment of sf-logging requires a service account identity, but one was not provided.

With only roles/apigee.admin you cannot yet create or actAs a service account, so the chassis ships with SA-free structured logging via AM-LogContext (which populates the audit.log flow variable, visible in trace and analytics) and keeps ML-CloudLogging on the shelf. To enable it after IAM elevation, deploy sf-logging with the -s identity flag:

enabling CloudLogging — after IAM elevationbash
# 1. Create a runtime SA and grant it logging access.
gcloud iam service-accounts create apigee-runtime \
  --project your-apigee-project
gcloud projects add-iam-policy-binding your-apigee-project \
  --member "serviceAccount:apigee-runtime@your-apigee-project.iam.gserviceaccount.com" \
  --role roles/logging.logWriter

# 2. Move ML-CloudLogging back into the bundle, add it to the manifest
#    and to sharedflows/default.xml, then deploy WITH the -s identity:
apigeecli sharedflows deploy -n sf-logging -e eval --ovr --wait \
  -s apigee-runtime@your-apigee-project.iam.gserviceaccount.com \
  -o your-apigee-project -t "$(gcloud auth print-access-token)"
The SA requirement is per-revision, not per-org

The identity is needed because the deployed revision contains a Cloud-integrated policy — it is not a one-time org setting. Any shared flow or proxy you later give a CloudLogging, GoogleAuthentication, or similar policy will demand the same -s identity at deploy time. The local deploy.sh passes no -s, which is exactly why ML-CloudLogging stays out of the default bundle. See CI/CD & IAM for the elevation path.
