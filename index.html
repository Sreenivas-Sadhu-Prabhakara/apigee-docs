Apigee X · Open Banking chassis

Overview

This is a policy chassis for Apigee X — a fleet of ten versioned shared flows that together implement the FAPI 1.0 Advanced security profile for UK Open Banking. API proxies stay deliberately thin: they declare a base path, route to a target, and delegate every cross-cutting concern — admission control, identity, message integrity, traffic, observability, and fault shaping — to the shared flows through FlowCallout steps. A team shipping a new API inherits the whole security posture by wiring in callouts; they write almost no security code themselves.

This site documents what the chassis actually does, flow by flow, exactly as it is built in the private code repository — real policy names, real step order, real conditions.

The policy-chassis pattern

A conventional proxy accretes security logic until it becomes unmaintainable and every team reinvents the same controls slightly differently. This chassis inverts that. Cross-cutting policy lives once, in shared flows; proxies compose them. The reference proxy bank-accounts-v1 is the canonical example — its entire PreFlow is a list of callouts, and the only business logic is a single mock-response step.

bank-accounts-v1/apiproxy/proxies/default.xmlXML
<PreFlow name="PreFlow">
  <Request>
    <Step><Name>FC-Preflight</Name></Step>
    <Step><Name>FC-FapiHeaders</Name></Step>
    <Step><Name>FC-JwtValidation</Name></Step>
    <Step>
      <Name>FC-JwsValidation</Name>
      <Condition>(request.header.x-jws-signature != null)</Condition>
    </Step>
    <!-- FC-MtlsBinding, FC-Jwe (both conditional), FC-TrafficManagement -->
  </Request>
</PreFlow>

Each FC-* step is a FlowCallout that invokes one shared flow. Because the proxy references shared flows by name, the chassis can be hardened in one place and every proxy inherits the change on the next deploy.

Conditions are the chassis logic

Several steps run only when relevant. Signature verification fires only when x-jws-signature is present; certificate binding only when the token carries a cnf claim or a client cert is forwarded; JWE decryption only for application/jose / application/jwe bodies. This keeps the demo GET flowing while leaving each control ready to be made mandatory per proxy.

Request lifecycle

Every request to an Apigee proxy traverses the same ordered pipeline. The PreFlow enforces the security chassis, the matched flow returns data, and the PostFlow finishes the response. Any thrown fault short-circuits to the DefaultFaultRule.

PreFlow · FC-Preflight

Admission control: per-IP spike arrest, then reject unsupported HTTP verbs (405), missing/invalid content types on body verbs (415), and payloads over 1 MiB (413).

PreFlow · FC-FapiHeaders

Validate and normalise FAPI headers — echo or mint x-fapi-interaction-id as a UUID, and validate x-fapi-customer-ip-address (IPv4/IPv6) when present.

PreFlow · FC-JwtValidation

Extract the bearer token, fault 401 if absent, verify the access-token JWT against the JWKS (signature, issuer, audience, expiry), and fault 403 without an accounts scope.

PreFlow · FC-JwsValidation conditional

Runs only when x-jws-signature is present. Verifies the detached PS256 request signature for non-repudiation of the message.

PreFlow · FC-MtlsBinding conditional

Runs when the token carries a cnf claim or a x-client-cert header is forwarded. Confirms the access token is bound to the presenting client certificate (cnf x5t#S256); fault on mismatch.

PreFlow · FC-Jwe conditional

Runs for application/jose / application/jwe content types. Decrypts the inbound JWE payload so downstream policy sees cleartext.

PreFlow · FC-TrafficManagement

Per-client spike arrest plus a per-client quota — protects the backend after the request has proven itself authentic and well-formed.

Route · GetAccounts → AM-MockAccounts

When proxy.pathsuffix matches /accounts on a GET, an AssignMessage returns a mock OB Account list. The proxy ships with RouteRule name="no-target" — no external dependency.

PostFlow · FC-SecurityHeaders

Stamp hardening response headers on the way out — HSTS, CSP, nosniff, and cache-control directives.

PostFlow · FC-Logging

Emit a structured log/analytics context record keyed by the interaction id for end-to-end traceability.

On any fault · FC-ErrorHandling

The DefaultFaultRule maps any raised fault to an OB / RFC 7807 error body with PII masking — so even a rejected request returns a clean, standard shape.

Order is load-bearing

Identity is established before message integrity, binding, and decryption; traffic management runs last in the PreFlow so quota is only spent on requests that have already passed authentication and admission control. Re-ordering these steps weakens the FAPI guarantees.

Why FAPI, why shared flows

Why FAPI 1.0 Advanced

UK Open Banking mandates the Financial-grade API profile. Beyond bearer tokens it requires sender-constrained tokens (so a stolen token is useless without the bound client certificate), detached request signatures for non-repudiation, and standardised error semantics. The chassis implements each of these as a discrete, testable control rather than as scattered proxy logic.

Why shared flows

Security written once is security audited once. Versioned shared flows give a single source of truth for every control, uniform enforcement across proxies, and a clean upgrade path — harden the chassis and every proxy inherits it. New APIs onboard by composing callouts, not by copy-pasting policy.
